## <a name="what-is-resiliency"></a><span data-ttu-id="26042-116">復原是什麼？</span><span class="sxs-lookup"><span data-stu-id="26042-116">What is resiliency?</span></span>

<span data-ttu-id="26042-120">復原的兩個重要層面是高可用性和災害復原。</span><span class="sxs-lookup"><span data-stu-id="26042-120">Two important aspects of resiliency are high availability and disaster recovery.</span></span>

* <span data-ttu-id="26042-121">**高可用性** (HA)：可讓應用程式繼續在狀況良好狀態下執行，而不需要長期停機。</span><span class="sxs-lookup"><span data-stu-id="26042-121">**High availability** (HA) is the ability of the application to continue running in a healthy state, without significant downtime.</span></span> <span data-ttu-id="26042-122">「狀況良好狀態」是指應用程式有回應，而且使用者可以連線到應用程式並與其互動。</span><span class="sxs-lookup"><span data-stu-id="26042-122">By "healthy state," we mean the application is responsive, and users can connect to the application and interact with it.</span></span>

<span data-ttu-id="26042-125">思考 HA 與 DR 的方法之一，是當錯誤的影響超過 HA 設計可處理錯誤的能力時即啟動 DR。</span><span class="sxs-lookup"><span data-stu-id="26042-125">One way to think about HA versus DR is that DR starts when the impact of a fault exceeds the ability of the HA design to handle it.</span></span>  

<span data-ttu-id="26042-126">在設計復原功能時，您必須了解可用性需求。</span><span class="sxs-lookup"><span data-stu-id="26042-126">When you design resiliency, you must understand your availability requirements.</span></span> <span data-ttu-id="26042-127">可接受多少停機時間？</span><span class="sxs-lookup"><span data-stu-id="26042-127">How much downtime is acceptable?</span></span> <span data-ttu-id="26042-128">這是有一部分是成本的功能。</span><span class="sxs-lookup"><span data-stu-id="26042-128">This is partly a function of cost.</span></span> <span data-ttu-id="26042-129">可能的停機時間會造成多少業務成本？</span><span class="sxs-lookup"><span data-stu-id="26042-129">How much will potential downtime cost your business?</span></span> <span data-ttu-id="26042-130">您應該將多少成本投資於此應用程式的高可用性？</span><span class="sxs-lookup"><span data-stu-id="26042-130">How much should you invest in making the application highly available?</span></span> <span data-ttu-id="26042-131">您也必須定義應用程式可用的意義。</span><span class="sxs-lookup"><span data-stu-id="26042-131">You also have to define what it means for the application to be available.</span></span> <span data-ttu-id="26042-132">例如，如果客戶可以提交訂單，但系統無法在一般時間範圍內處理，則應用程式是否為「停機」？</span><span class="sxs-lookup"><span data-stu-id="26042-132">For example, is the application "down" if a customer can submit an order but the system cannot process it within the normal timeframe?</span></span> <span data-ttu-id="26042-133">也請考慮特定類型中斷的發生機率，以及因應措施是否符合成本效益。</span><span class="sxs-lookup"><span data-stu-id="26042-133">Also consider the probability of a particular type of outage occurring, and whether a mitigation strategy is cost-effective.</span></span>

<span data-ttu-id="26042-134">另一個常見的字詞是**業務續航力** (BC)，這是指在不利的狀況 (例如天然災害或服務暫停) 期間和之後，執行重要商務功能的能力。</span><span class="sxs-lookup"><span data-stu-id="26042-134">Another common term is **business continuity** (BC), which is the ability to perform essential business functions during and after adverse conditions, such as a natural disaster or a downed service.</span></span> <span data-ttu-id="26042-135">BC 涵蓋整個企業的作業，包括實體設備、人員、通訊、運輸和 IT。</span><span class="sxs-lookup"><span data-stu-id="26042-135">BC covers the entire operation of the business, including physical facilities, people, communications, transportation, and IT.</span></span> <span data-ttu-id="26042-136">本文著重於雲端應用程式，但必須在整體 BC 需求的內容中完成復原規劃。</span><span class="sxs-lookup"><span data-stu-id="26042-136">This article focuses on cloud applications, but resilience planning must be done in the context of overall BC requirements.</span></span> 

<span data-ttu-id="26042-137">**資料備份**是 DR 的重要部分。</span><span class="sxs-lookup"><span data-stu-id="26042-137">**Data backup** is a critical part of DR.</span></span> <span data-ttu-id="26042-138">如果應用程式的無狀態元件失敗，您隨時都可加以重新佈署。</span><span class="sxs-lookup"><span data-stu-id="26042-138">If the stateless components of an application fail, you can always redeploy them.</span></span> <span data-ttu-id="26042-139">但如果資料遺失，系統就無法回復穩定狀態。</span><span class="sxs-lookup"><span data-stu-id="26042-139">But if data is lost, the system can't return to a stable state.</span></span> <span data-ttu-id="26042-140">您必須備份資料，而且為了免於全區域災害，最好是備份到不同區域。</span><span class="sxs-lookup"><span data-stu-id="26042-140">Data must be backed up, ideally in a different region in case of a region-wide disaster.</span></span> 

<span data-ttu-id="26042-141">備份與**資料複寫**不同。</span><span class="sxs-lookup"><span data-stu-id="26042-141">Backup is distinct from **data replication**.</span></span> <span data-ttu-id="26042-142">資料複寫會近乎即時地複製資料，讓系統可以快速容錯移轉至複本。</span><span class="sxs-lookup"><span data-stu-id="26042-142">Data replication involves copying data in near-real-time, so that the system can fail over quickly to a replica.</span></span> <span data-ttu-id="26042-143">許多資料庫系統都支援複寫，例如，SQL Server 就支援 SQL Server Always On 可用性群組。</span><span class="sxs-lookup"><span data-stu-id="26042-143">Many databases systems support replication; for example, SQL Server supports SQL Server Always On Availability Groups.</span></span> <span data-ttu-id="26042-144">資料複寫可藉由確保資料複本隨時做好準備，來減少從中斷復原所需的時間長度。</span><span class="sxs-lookup"><span data-stu-id="26042-144">Data replication can reduce how long it takes to recover from an outage, by ensuring that a replica of the data is always standing by.</span></span> <span data-ttu-id="26042-145">不過，資料複寫無法防範人為錯誤。</span><span class="sxs-lookup"><span data-stu-id="26042-145">However, data replication won't protect against human error.</span></span> <span data-ttu-id="26042-146">如果資料因為人為錯誤而損毀，損毀的資料就會複製到複本中。</span><span class="sxs-lookup"><span data-stu-id="26042-146">If data gets corrupted because of human error, the corrupted data just gets copied to the replicas.</span></span> <span data-ttu-id="26042-147">因此，您仍需要在 DR 策略中包含長期備份措施。</span><span class="sxs-lookup"><span data-stu-id="26042-147">Therefore, you still need to include long-term backup in your DR strategy.</span></span>

* <span data-ttu-id="26042-524">**警示**。</span><span class="sxs-lookup"><span data-stu-id="26042-524">**Alerts**.</span></span> <span data-ttu-id="26042-525">監視您的應用程式，注意可能需要主動介入的跡象。</span><span class="sxs-lookup"><span data-stu-id="26042-525">Monitor your application for warning signs that may require proactive intervention.</span></span> <span data-ttu-id="26042-526">例如，如果您看見 SQL Database 或 Cosmos DB 一致地節流您的應用程式，可能就需要增加您的資料庫容量或最佳化查詢。</span><span class="sxs-lookup"><span data-stu-id="26042-526">For example, if you see that SQL Database or Cosmos DB consistently throttles your application, you might need to increase your database capacity or optimize your queries.</span></span> <span data-ttu-id="26042-527">在此範例中，即使應用程式可能會以透明的方式節流錯誤，您的遙測仍應發出警示，讓您可以追蹤。</span><span class="sxs-lookup"><span data-stu-id="26042-527">In this example, even though the application might handle the throttling errors transparently, your telemetry should still raise an alert so that you can follow up.</span></span>  
* <span data-ttu-id="26042-528">**手動容錯移轉**。</span><span class="sxs-lookup"><span data-stu-id="26042-528">**Manual failover**.</span></span> <span data-ttu-id="26042-529">某些系統無法自動容錯移轉，而需要手動容錯移轉。</span><span class="sxs-lookup"><span data-stu-id="26042-529">Some systems cannot fail over automatically and require a manual failover.</span></span> 
* <span data-ttu-id="26042-530">**作業整備測試**。</span><span class="sxs-lookup"><span data-stu-id="26042-530">**Operational readiness testing**.</span></span> <span data-ttu-id="26042-531">如果您的應用程式容錯移轉到次要地區，就應該在容錯回復到主要區域之前，執行作業整備測試。</span><span class="sxs-lookup"><span data-stu-id="26042-531">If your application fails over to a secondary region, you should perform an operational readiness test before you fail back to the primary region.</span></span> <span data-ttu-id="26042-532">測試應該確認主要區域健康情況，且就緒可再次接收流量。</span><span class="sxs-lookup"><span data-stu-id="26042-532">The test should verify that the primary region is healthy and ready to receive traffic again.</span></span>
* <span data-ttu-id="26042-533">**資料一致性檢查**。</span><span class="sxs-lookup"><span data-stu-id="26042-533">**Data consistency check**.</span></span> <span data-ttu-id="26042-534">如果失敗發生在資料存放區中，當存放區再次可用時，可能會有資料不一致的情況，特別是在已複寫資料時。</span><span class="sxs-lookup"><span data-stu-id="26042-534">If a failure happens in a data store, there may be data inconsistencies when the store becomes available again, especially if the data was replicated.</span></span>

<span data-ttu-id="26042-537">記錄並測試災害復原計畫。</span><span class="sxs-lookup"><span data-stu-id="26042-537">Document and test your disaster recovery plan.</span></span> <span data-ttu-id="26042-538">評估應用程式失敗的商務影響。</span><span class="sxs-lookup"><span data-stu-id="26042-538">Evaluate the business impact of application failures.</span></span> <span data-ttu-id="26042-539">盡可能自動化程序並記錄任何手動步驟，例如手動容錯移轉或從備份還原資料。</span><span class="sxs-lookup"><span data-stu-id="26042-539">Automate the process as much as possible, and document any manual steps, such as manual failover or data restoration from backups.</span></span> <span data-ttu-id="26042-540">定期測試您的災害復原程序來驗證和改善計劃。</span><span class="sxs-lookup"><span data-stu-id="26042-540">Regularly test your disaster recovery process to validate and improve the plan.</span></span>
