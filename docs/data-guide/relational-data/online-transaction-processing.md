# <a name="online-transaction-processing-oltp"></a><span data-ttu-id="441c9-102">線上交易處理 (OLTP)</span><span class="sxs-lookup"><span data-stu-id="441c9-102">Online transaction processing (OLTP)</span></span>

## <a name="transactional-data"></a><span data-ttu-id="441c9-105">交易資料</span><span class="sxs-lookup"><span data-stu-id="441c9-105">Transactional data</span></span>

<span data-ttu-id="441c9-106">交易資料是追蹤組織活動的相關互動所產生的資訊。</span><span class="sxs-lookup"><span data-stu-id="441c9-106">Transactional data is information that tracks the interactions related to an organization's activities.</span></span> <span data-ttu-id="441c9-107">這些互動通常是商業交易，例如，客戶的付款、對供應商的付款、庫存的產品、接到的訂單，或是交付的服務。</span><span class="sxs-lookup"><span data-stu-id="441c9-107">These interactions are typically business transactions, such as payments received from customers, payments made to suppliers, products moving through inventory, orders taken, or services delivered.</span></span> <span data-ttu-id="441c9-108">交易事件代表交易本身，通常會包含時間維度、一些數值，以及對其他資料的參考。</span><span class="sxs-lookup"><span data-stu-id="441c9-108">Transactional events, which represent the transactions themselves, typically contain a time dimension, some numerical values, and references to other data.</span></span>

<span data-ttu-id="441c9-109">交易通常必須是*不可部分完成*且*一致*的。</span><span class="sxs-lookup"><span data-stu-id="441c9-109">Transactions typically need to be *atomic* and *consistent*.</span></span> <span data-ttu-id="441c9-110">不可部分完成性表示整個交易無論成功或失敗都一律是工作單位整體的，且絕不會處於半完成的狀態。</span><span class="sxs-lookup"><span data-stu-id="441c9-110">Atomicity means that an entire transaction always succeeds or fails as one unit of work, and is never left in a half-completed state.</span></span> <span data-ttu-id="441c9-111">如果無法完成交易，則資料庫系統必須復原該交易中已完成的任何步驟。</span><span class="sxs-lookup"><span data-stu-id="441c9-111">If a transaction cannot be completed, the database system must roll back any steps that were already done as part of that transaction.</span></span> <span data-ttu-id="441c9-112">在傳統的 RDBMS 中，只要交易無法完成，就會自動執行此復原。</span><span class="sxs-lookup"><span data-stu-id="441c9-112">In a traditional RDBMS, this rollback happens automatically if a transaction cannot be completed.</span></span> <span data-ttu-id="441c9-113">一致性表示交易一律會讓資料保持在有效狀態。</span><span class="sxs-lookup"><span data-stu-id="441c9-113">Consistency means that transactions always leave the data in a valid state.</span></span> <span data-ttu-id="441c9-114">(以上關於不可部分完成性與一致性的說明只是口語化的。</span><span class="sxs-lookup"><span data-stu-id="441c9-114">(These are very informal descriptions of atomicity and consistency.</span></span> <span data-ttu-id="441c9-115">這些屬性另有更為正式的定義，例如 [ACID](https://en.wikipedia.org/wiki/ACID)。)</span><span class="sxs-lookup"><span data-stu-id="441c9-115">There are more formal definitions of these properties, such as [ACID](https://en.wikipedia.org/wiki/ACID).)</span></span>

<span data-ttu-id="441c9-116">交易資料庫可對使用各種不同鎖定策略 (例如封閉式鎖定) 的交易支援強式一致性，以確保企業內容中的所有資料對於所有使用者和程序而言都絕對是一致的。</span><span class="sxs-lookup"><span data-stu-id="441c9-116">Transactional databases can support strong consistency for transactions using various locking strategies, such as pessimistic locking, to ensure that all data is strongly consistent within the context of the enterprise, for all users and processes.</span></span>

<span data-ttu-id="441c9-117">使用交易資料最常見的部署架構，是 3 層式架構中的資料存放區層。</span><span class="sxs-lookup"><span data-stu-id="441c9-117">The most common deployment architecture that uses transactional data is the data store tier in a 3-tier architecture.</span></span> <span data-ttu-id="441c9-118">3 層式架構通常由展示層、商務邏輯層和資料存放區層所組成。</span><span class="sxs-lookup"><span data-stu-id="441c9-118">A 3-tier architecture typically consists of a presentation tier, business logic tier, and data store tier.</span></span> <span data-ttu-id="441c9-119">相關部署架構為[多層式](/azure/architecture/guide/architecture-styles/n-tier)架構，此類架構可能會以多個中介層來處理商務邏輯。</span><span class="sxs-lookup"><span data-stu-id="441c9-119">A related deployment architecture is the [N-tier](/azure/architecture/guide/architecture-styles/n-tier) architecture, which may have multiple middle-tiers handling business logic.</span></span>

## <a name="when-to-use-this-solution"></a><span data-ttu-id="441c9-154">此解決方案的使用時機</span><span class="sxs-lookup"><span data-stu-id="441c9-154">When to use this solution</span></span>

<span data-ttu-id="441c9-157">OLTP 系統是設計成能夠有效率地處理和儲存交易，以及查詢交易資料。</span><span class="sxs-lookup"><span data-stu-id="441c9-157">OLTP systems are designed to efficiently process and store transactions, as well as query transactional data.</span></span> <span data-ttu-id="441c9-158">若要讓 OLTP 系統有效率地處理並儲存個別交易，部分需藉助資料正規化來達成 &mdash;，也就是將資料分解成較不重複的較小區塊。</span><span class="sxs-lookup"><span data-stu-id="441c9-158">The goal of efficiently processing and storing individual transactions by an OLTP system is partly accomplished by data normalization &mdash; that is, breaking the data up into smaller chunks that are less redundant.</span></span> <span data-ttu-id="441c9-159">如此會提升效率，因為它可讓 OLTP 系統獨立處理大量交易，並避免為維護資料完整性所需的額外處理 (如果存在重複的資料)。</span><span class="sxs-lookup"><span data-stu-id="441c9-159">This supports efficiency because it enables the OLTP system to process large numbers of transactions independently, and avoids extra processing needed to maintain data integrity in the presence of redundant data.</span></span>

## <a name="challenges"></a><span data-ttu-id="441c9-160">挑戰</span><span class="sxs-lookup"><span data-stu-id="441c9-160">Challenges</span></span>

<span data-ttu-id="441c9-161">實作和使用 OLTP 系統可能產生幾個挑戰：</span><span class="sxs-lookup"><span data-stu-id="441c9-161">Implementing and using an OLTP system can create a few challenges:</span></span>

- <span data-ttu-id="441c9-162">OLTP 系統並不一定適合用來處理大量資料的彙總，但有例外狀況，例如規劃良好的 SQL Server 型解決方案。</span><span class="sxs-lookup"><span data-stu-id="441c9-162">OLTP systems are not always good for handling aggregates over large amounts of data, although there are exceptions, such as a well-planned SQL Server-based solution.</span></span> <span data-ttu-id="441c9-163">對 OLTP 系統而言，針對資料 (依賴超過數百萬筆個別交易的彙總計算) 進行分析相當耗費資源。</span><span class="sxs-lookup"><span data-stu-id="441c9-163">Analytics against the data, that rely on aggregate calculations over millions of individual transactions, are very resource intensive for an OLTP system.</span></span> <span data-ttu-id="441c9-164">它們可能會很慢的執行，並可能因封鎖資料庫的其他交易而導致速度變慢。</span><span class="sxs-lookup"><span data-stu-id="441c9-164">They can be slow to execute and can cause a slow-down by blocking other transactions in the database.</span></span>
- <span data-ttu-id="441c9-165">進行分析和報告高度正規化的資料時，查詢通常會很複雜，因為大多數的查詢需要使用聯結來解除資料的正規化。</span><span class="sxs-lookup"><span data-stu-id="441c9-165">When conducting analytics and reporting on data that is highly normalized, the queries tend to be complex, because most queries need to de-normalize the data by using joins.</span></span> <span data-ttu-id="441c9-166">此外，OLTP 系統中資料庫物件的命名慣例通常要求簡單易讀。</span><span class="sxs-lookup"><span data-stu-id="441c9-166">Also, naming conventions for database objects in OLTP systems tend to be terse and succinct.</span></span> <span data-ttu-id="441c9-167">增加的正規化搭配簡易的命名慣例，使得商務使用者很難在不求助 DBA 或資料開發人員的協助下，使用 OLTP 系統進行查詢。</span><span class="sxs-lookup"><span data-stu-id="441c9-167">The increased normalization coupled with terse naming conventions makes OLTP systems difficult for business users to query, without the help of a DBA or data developer.</span></span>
- <span data-ttu-id="441c9-168">無限期儲存交易記錄，以及在任何一個資料表中儲存太多資料時，可能會導致查詢效能下降 (根據所儲存的交易數目而定)。</span><span class="sxs-lookup"><span data-stu-id="441c9-168">Storing the history of transactions indefinitely and storing too much data in any one table can lead to slow query performance, depending on the number of transactions stored.</span></span> <span data-ttu-id="441c9-169">常見的解決方案是在 OLTP 系統中保留一段相關的時段 (例如目前的會計年度)，並將歷史資料卸載至其他系統，例如資料超市或[資料倉儲](./data-warehousing.md)。</span><span class="sxs-lookup"><span data-stu-id="441c9-169">The common solution is to maintain a relevant window of time (such as the current fiscal year) in the OLTP system and offload historical data to other systems, such as a data mart or [data warehouse](./data-warehousing.md).</span></span>
