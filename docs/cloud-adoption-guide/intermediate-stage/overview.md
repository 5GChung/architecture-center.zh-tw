## <a name="section-1-azure-concepts-for-multiple-workloads-and-multiple-teams"></a><span data-ttu-id="78250-121">第 1 節：多個工作負載和多個小組的 Azure 概念</span><span class="sxs-lookup"><span data-stu-id="78250-121">Section 1: Azure concepts for multiple workloads and multiple teams</span></span>

<span data-ttu-id="78250-122">在基礎採用階段中，您已了解有關 Azure 內部與如何建立、讀取、更新和刪除資源的一些基本概念。</span><span class="sxs-lookup"><span data-stu-id="78250-122">In the foundational adoption stage, you learned some basics about Azure internals and how resources are created, read, updated, and deleted.</span></span> <span data-ttu-id="78250-123">您也了解身分識別，且 Azure 只信任 Azure Active Directory (AD) 驗證及授權需要這些資源存取權的使用者。</span><span class="sxs-lookup"><span data-stu-id="78250-123">You also learned about identity and that Azure only trusts Azure Active Directory (AD) to authenticate and authorize users who need access to those resources.</span></span>

<span data-ttu-id="78250-124">您也會開始了解如何設定 Azure 的治理工具，來管理貴組織對於 Azure 資源的使用。</span><span class="sxs-lookup"><span data-stu-id="78250-124">You also started learning about how to configure Azure's governance tools to manage your organization's use of Azure resources.</span></span> <span data-ttu-id="78250-125">在基礎階段中我們討論了如何治理單一小組部署簡單工作負載所需的資源存取權。</span><span class="sxs-lookup"><span data-stu-id="78250-125">In the foundational stage we looked at how to govern a single team's access to the resources necessary to deploy a simple workload.</span></span> <span data-ttu-id="78250-126">事實上，貴組織會組成多個小組，同時處理多個工作負載。</span><span class="sxs-lookup"><span data-stu-id="78250-126">In reality, your organization is going to be made up of multiple teams working on multiple workloads simultaneously.</span></span> 

<span data-ttu-id="78250-127">開始之前，讓我們看看**工作負載**這個詞彙真正的意義。</span><span class="sxs-lookup"><span data-stu-id="78250-127">Before we begin, let's take a look at what the term **workload** actually means.</span></span> <span data-ttu-id="78250-128">這個詞彙通常被理解為定義任意單位的功能，例如應用程式或服務。</span><span class="sxs-lookup"><span data-stu-id="78250-128">It's a term that is typically understood to define an arbitrary unit of functionality such as an application or service.</span></span> <span data-ttu-id="78250-129">我們會以部署到伺服器以及所需任何其他服務 (例如資料庫) 的程式碼成品角度來思考工作負載。</span><span class="sxs-lookup"><span data-stu-id="78250-129">We think about a workload in terms of the code artifacts that are deployed to a server as well as any other services, such as a database, that are necessary.</span></span> <span data-ttu-id="78250-130">這對於內部部署應用程式或服務是有用的定義，但是在雲端中我們就需要加以延伸。</span><span class="sxs-lookup"><span data-stu-id="78250-130">This is a useful definition for an on-premises application or service but in the cloud we need to expand on it.</span></span> 

<span data-ttu-id="78250-131">在雲端中，工作負載不僅是包括所有成品，同時也包含雲端資源。</span><span class="sxs-lookup"><span data-stu-id="78250-131">In the cloud, a workload not only encompasses all the artifacts but also includes the cloud resources as well.</span></span> <span data-ttu-id="78250-132">我們包含雲端資源作為定義的一部分，是因為所謂**基礎結構即程式碼**的概念。</span><span class="sxs-lookup"><span data-stu-id="78250-132">We include cloud resources as part of our definition because of a concept known as **infrastructure-as-code**.</span></span> <span data-ttu-id="78250-133">如同您在「Azure 的運作方式」說明中所了解的，Azure 中的資源是由協調器服務部署。</span><span class="sxs-lookup"><span data-stu-id="78250-133">As you learned in the "how does Azure work" explainer, resources in Azure are deployed by an orchestrator service.</span></span> <span data-ttu-id="78250-134">協調器服務會透過 Web API 公開這個功能，此 Web API 可以使用數個工具來呼叫，例如 Powershell、Azure 命令列介面 (CLI) 以及 Azure 入口網站。</span><span class="sxs-lookup"><span data-stu-id="78250-134">The orchestrator service exposes this functionality through a web API, and this web API can be called using several tools such as Powershell, the Azure command line interface (CLI), and the Azure portal.</span></span> <span data-ttu-id="78250-135">這表示我們可以在電腦可讀取檔案中指定我們的資源，該檔案可以與程式碼成品 (與我們的應用程式相關聯) 一起儲存。</span><span class="sxs-lookup"><span data-stu-id="78250-135">This means that we can specify our resources in a machine-readable file that can be stored along with the code artifacts associated with our application.</span></span>

<span data-ttu-id="78250-136">可以讓我們以程式碼成品和必要雲端資源的角度定義工作負載，進一步讓我們**隔離**我們的工作負載。</span><span class="sxs-lookup"><span data-stu-id="78250-136">This enables us to define a workload in terms of code artifacts and the necessary cloud resources, and this further enables us to **isolate** our workloads.</span></span> <span data-ttu-id="78250-137">工作負載可能會依據資源的組織方式、網路拓撲或其他屬性，而遭到隔離。</span><span class="sxs-lookup"><span data-stu-id="78250-137">Workloads may be isolated by the way resources are organized, by network topology, or by other attributes.</span></span> <span data-ttu-id="78250-138">工作負載隔離的目標是要將工作負載的特定資源關聯至小組，讓小組可以獨立管理這些資源的所有層面。</span><span class="sxs-lookup"><span data-stu-id="78250-138">The goal of workload isolation is to associate a workload's specific resources to a team so the team can independently manage all aspects of those resources.</span></span> <span data-ttu-id="78250-139">這可讓多個小組共用 Azure 中的資源管理服務，同時防止不小心刪除或修改彼此的資源。</span><span class="sxs-lookup"><span data-stu-id="78250-139">This enables multiple teams to share resource management services in Azure while preventing the unintentional deletion or modification of each other's resources.</span></span>

<span data-ttu-id="78250-140">這個隔離也會啟用稱為 [DevOps](https://azure.microsoft.com/solutions/devops/) 的另一個概念。</span><span class="sxs-lookup"><span data-stu-id="78250-140">This isolation also enables another concept known as [DevOps](https://azure.microsoft.com/solutions/devops/).</span></span> <span data-ttu-id="78250-141">DevOps 包含軟體開發實務，其中包含上述的軟體開發和 IT 作業，但是盡可能新增自動化的使用。</span><span class="sxs-lookup"><span data-stu-id="78250-141">DevOps includes the software development practices that include both software development and IT operations above, but adds the use of automation as much as possible.</span></span> <span data-ttu-id="78250-142">DevOps 的其中一個原則稱為持續整合與持續傳遞 (CI/CD)。</span><span class="sxs-lookup"><span data-stu-id="78250-142">One of the principles of DevOps is known as continuous integration and continuous delivery (CI/CD).</span></span> <span data-ttu-id="78250-143">持續整合是指開發人員認可程式碼變更時每次都會執行的自動化建置程序，而持續傳遞是指將這個程式碼部署到各種**環境** (例如進行測試的**開發環境**或進行最終部署的**生產環境**) 的自動化程序。</span><span class="sxs-lookup"><span data-stu-id="78250-143">Continuous integration refers to the automated build processes that are run each time a developer commits a code change, and continuous delivery refers to the automated processes that deploy this code to various **environments** such as a **development environment** for testing or a **production environment** for final deployment.</span></span>
