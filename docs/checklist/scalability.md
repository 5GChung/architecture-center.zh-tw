## <a name="service-design"></a><span data-ttu-id="e7f11-104">服務設計</span><span class="sxs-lookup"><span data-stu-id="e7f11-104">Service design</span></span>
* <span data-ttu-id="e7f11-105">**分割工作負載**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-105">**Partition the workload**.</span></span> <span data-ttu-id="e7f11-106">將程序的組件設計為離散與可分解狀態。</span><span class="sxs-lookup"><span data-stu-id="e7f11-106">Design parts of the process to be discrete and decomposable.</span></span> <span data-ttu-id="e7f11-107">在遵循分離需求功能的一般規則與單一責任原則的情況下，將各個組件的大小減至最小。</span><span class="sxs-lookup"><span data-stu-id="e7f11-107">Minimize the size of each part, while following the usual rules for separation of concerns and the single responsibility principle.</span></span> <span data-ttu-id="e7f11-108">如此一來，散佈元件組件時，即可充分運用每個計算單位 (例如角色或資料庫伺服器)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-108">This allows the component parts to be distributed in a way that maximizes use of each compute unit (such as a role or database server).</span></span> <span data-ttu-id="e7f11-109">它也能藉由新增特定資源的執行個體，更輕鬆地調整應用程式。</span><span class="sxs-lookup"><span data-stu-id="e7f11-109">It also makes it easier to scale the application by adding instances of specific resources.</span></span> <span data-ttu-id="e7f11-110">如需詳細資訊，請參閱「 [計算分割指引](https://msdn.microsoft.com/library/dn589773.aspx)」。</span><span class="sxs-lookup"><span data-stu-id="e7f11-110">For more information, see [Compute Partitioning Guidance](https://msdn.microsoft.com/library/dn589773.aspx).</span></span>
* <span data-ttu-id="e7f11-111">**調整設計**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-111">**Design for scaling**.</span></span> <span data-stu-id="e7f11-113">However, the application must be designed with this in mind.</span></span> <span data-ttu-id="e7f11-114">例如，該應用程式及其使用的服務必須處於無狀態，才能將要求路由傳送到任何執行個體。</span><span class="sxs-lookup"><span data-stu-id="e7f11-114">For example, the application and the services it uses must be stateless, to allow requests to be routed to any instance.</span></span> <span data-ttu-id="e7f11-115">這可避免在新增或移除特定執行個體時，對目前的使用者造成不良的影響。</span><span class="sxs-lookup"><span data-stu-id="e7f11-115">This also prevents the addition or removal of specific instances from adversely Web 應用程式可能會以循環配置資源的方式使用一組佇列，將要求路由至以背景工作角色執行的背景服務。</span><span class="sxs-lookup"><span data-stu-id="e7f11-117">For example, a web application might use a set of queues in a round-robin approach to data-stu-id="e7f11-118">The web application must be able to detect changes in the number of queues, to successfully route requests and balance the load on the application.</span></span>
* <span data-ttu-id="e7f11-119">**以單位進行調整**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-119">**Scale as a unit**.</span></span> <span data-ttu-id="e7f11-120">規劃其他資源，以因應成長。</span><span class="sxs-lookup"><span data-stu-id="e7f11-120">Plan for additional resources to accommodate growth.</span></span> <span data-ttu-id="e7f11-121">了解每個資源的調整上限，並利用分區化或分解方式來突破這些限制。</span><span class="sxs-lookup"><span data-stu-id="e7f11-121">For each resource, know the upper scaling limits, and use sharding or decomposition to go beyond these limits.</span></span> <span data-ttu-id="e7f11-122">根據定義完善的資源組決定系統縮放單位。</span><span class="sxs-lookup"><span data-stu-id="e7f11-122">Determine the scale units for the system in terms of well-defined sets of resources.</span></span> <span data-ttu-id="e7f11-123">在整個系統的某部分缺乏資源所加諸的限制下，這樣能夠更輕鬆地套用相應放大作業，並對應用程式產生較少的負面影響。</span><span class="sxs-lookup"><span data-stu-id="e7f11-123">This makes applying scale-out operations easier, and less prone to negative impact on the application through limitations imposed by lack of resources in some part of the overall system.</span></span> <span data-ttu-id="e7f11-124">例如，新增 x 個 Web 和背景工作角色可能需要 y 個額外佇列和 z 個儲存體帳戶，以處理這些角色所產生的額外工作負載。</span><span class="sxs-lookup"><span data-stu-id="e7f11-124">For example, adding x number of web and worker roles might require y number of additional queues and z number of storage accounts to handle the additional workload generated by the roles.</span></span> <span data-ttu-id="e7f11-125">因此，一個縮放單位可能包含 x 個 Web 和背景工作角色、y 個佇列和 z 個儲存體帳戶。</span><span class="sxs-lookup"><span data-stu-id="e7f11-125">So a scale unit could consist of x web and worker roles, *y* queues, and *z* storage accounts.</span></span> <span data-ttu-id="e7f11-126">應用程式應設計為能藉由新增一或多個縮放單位來輕鬆調整。</span><span class="sxs-lookup"><span data-stu-id="e7f11-126">Design the application so that it's easily scaled by adding one or more scale units.</span></span>
* <span data-ttu-id="e7f11-127">**避免用戶端親和性**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-127">**Avoid client affinity**.</span></span> <span data-ttu-id="e7f11-128">可能的話，請確認應用程式不需親和性。</span><span class="sxs-lookup"><span data-stu-id="e7f11-128">Where possible, ensure that the application does not require affinity.</span></span> <span data-ttu-id="e7f11-129">因而可將要求路由傳送至任一個執行個體，而執行個體的數量並不重要。</span><span class="sxs-lookup"><span data-stu-id="e7f11-129">Requests can thus be routed to any instance, and the number of instances is irrelevant.</span></span> <span data-ttu-id="e7f11-130">這也可避免進行儲存、擷取與維護每位使用者狀態資訊所產生的額外負荷。</span><span class="sxs-lookup"><span data-stu-id="e7f11-130">This also avoids the overhead of storing, retrieving, and maintaining state information for each user.</span></span>
* <span data-ttu-id="e7f11-131">**利用平台自動調整功能**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-131">**Take advantage of platform autoscaling features**.</span></span> <span data-ttu-id="e7f11-132">通常會優先使用裝載平台支援的自動調整功能 (如 Azure 自動調整)，只有在內建機制無法滿足需求的情況下才會使用自訂或第三方機制 。</span><span class="sxs-lookup"><span data-stu-id="e7f11-132">Where the hosting platform supports an autoscaling capability, such as Azure Autoscale, prefer it to custom or third-party mechanisms unless the built-in mechanism can't fulfill your requirements.</span></span> <span data-ttu-id="e7f11-133">請儘可能使用排程的調整規則，確保資源可正常使用而不會造成啟動延遲，但適度地將重新啟動自動調整新增至規則中，以在必要時因應非預期的變更。</span><span class="sxs-lookup"><span data-stu-id="e7f11-133">Use scheduled scaling rules where possible to ensure resources are available without a start-up delay, but add reactive autoscaling to the rules where appropriate to cope with unexpected changes in demand.</span></span> <span data-ttu-id="e7f11-134">您可以使用服務管理 API 中的自動調整作業來調整自動調整功能，並將自訂計數器新增至規則中 。</span><span class="sxs-lookup"><span data-stu-id="e7f11-134">You can use the autoscaling operations in the Service Management API to adjust autoscaling, and to add custom counters to rules.</span></span> <span data-ttu-id="e7f11-135">如需詳細資訊，請參閱 [自動調整指引](../best-practices/auto-scaling.md)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-135">For more information, see [Auto-scaling guidance](../best-practices/auto-scaling.md).</span></span>
* <span data-ttu-id="e7f11-136">**卸載密集 CPU/IO 工作作為背景工作**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-136">**Offload intensive CPU/IO tasks as background tasks**.</span></span> <span data-ttu-id="e7f11-137">若預期對服務的要求需要較長的時間來執行或吸收大量資源，請卸載此要求的處理並轉為個別工作。</span><span class="sxs-lookup"><span data-stu-id="e7f11-137">If a request to a service is expected to take a long time to run or absorb considerable resources, offload the processing for this request to a separate task.</span></span> <span data-ttu-id="e7f11-138">利用背景工作角色或背景工作 (依裝載平台而定) 來執行這些工作。</span><span class="sxs-lookup"><span data-stu-id="e7f11-138">Use worker roles or background jobs (depending on the hosting platform) to execute these tasks.</span></span> <span data-ttu-id="e7f11-139">這項策略可讓服務繼續接收更多要求，並能保持正常回應。</span><span class="sxs-lookup"><span data-stu-id="e7f11-139">This strategy enables the service to continue receiving further requests and remain responsive.</span></span>  <span data-ttu-id="e7f11-140">如需詳細資訊，請參閱「 [背景工作指引](../best-practices/background-jobs.md)」(英文)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-140">For more information, see [Background jobs guidance](../best-practices/background-jobs.md).</span></span>
* <span data-ttu-id="e7f11-141">**分散背景工作的工作負載**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-141">**Distribute the workload for background tasks**.</span></span> <span data-ttu-id="e7f11-142">當背景工作很多，或工作需要大量時間或資源，請將工作分散到多個計算單位 (例如背景工作角色或背景工作)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-142">Where there are many background tasks, or the tasks require considerable time or resources, spread the work across multiple compute units (such as worker roles or background jobs).</span></span> <span data-ttu-id="e7f11-143">[Competing Consumers Pattern (競爭取用者模式)](https://msdn.microsoft.com/library/dn568101.aspx)提供了一個可能的解決方案。</span><span class="sxs-lookup"><span data-stu-id="e7f11-143">For one possible solution, see the [Competing Consumers Pattern](https://msdn.microsoft.com/library/dn568101.aspx).</span></span>
* <span data-ttu-id="e7f11-144">考慮採用「無共用」架構。</span><span class="sxs-lookup"><span data-stu-id="e7f11-144">**Consider moving towards a *shared-nothing* architecture**.</span></span> <span data-ttu-id="e7f11-145">無共用架構使用獨立且自給自足的節點，完全不會出現爭用情況 (例如，共用服務或儲存體)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-145">A shared-nothing architecture uses independent, self-sufficient nodes that have no single point of contention (such as shared services or storage).</span></span> <span data-ttu-id="e7f11-146">理論上，這類系統幾乎可以無限期地調整。</span><span class="sxs-lookup"><span data-stu-id="e7f11-146">In theory, such a system can scale almost indefinitely.</span></span> <span data-ttu-id="e7f11-147">雖然一般而言，完全無共用架構的方法對大部分的應用程式來說並不實用，卻可能因此有機會設計更好的延展性。</span><span class="sxs-lookup"><span data-stu-id="e7f11-147">While a fully shared-nothing approach is generally not practical for most applications, it may provide opportunities to design for better scalability.</span></span> <span data-ttu-id="e7f11-148">例如，避免使用伺服器端工作階段狀態、用戶端親和性與資料分割，都是採用無共用架構的絕佳範例。</span><span class="sxs-lookup"><span data-stu-id="e7f11-148">For example, avoiding the use of server-side session state, client affinity, and data partitioning are good examples of moving towards a shared-nothing architecture.</span></span>

## <a name="data-management"></a><span data-ttu-id="e7f11-149">資料管理</span><span class="sxs-lookup"><span data-stu-id="e7f11-149">Data management</span></span>
* <span data-ttu-id="e7f11-150">**使用資料分割**。</span><span class="sxs-lookup"><span data-stu-id="e7f11-150">**Use data partitioning**.</span></span> <span data-ttu-id="e7f11-151">將資料分割至多個資料庫和資料庫伺服器，或將應用程式設計為使用提供明確分割功能的資料儲存體服務 (範例包括 Azure SQL Database 彈性資料庫和 Azure 表格儲存體)。</span><span class="sxs-lookup"><span data-stu-id="e7f11-151">Divide the data across and Azure Table storage).</span></span> <span data-ttu-id="e7f11-152">此方法有助於達到最佳效能，使調整更加輕鬆自如。</span><span class="sxs-lookup"><span data-stu-id="e7f11-152">This approach can help to maximize performance and allow easier scaling.</span></span> <span data-ttu-id="e7f11-153">資料分割技術有許多種，例如水平資料分割、垂直資料分割與功能資料分割。</span><span class="sxs-lookup"><span data-stu-id="e7f11-153">There are different partitioning techniques, such as horizontal, vertical, and functional.</span></span> <span data-ttu-id="e7f11-154">您可以合併使用以達到最佳效能，上至提升的查詢效能、較簡單的延展性、更佳的管理靈活度、更高的可用性，下至比對存放區類型與其保存的資料。</span><span class="sxs-lookup"><span data-stu-id="e7f11-154">You can use a combination of these to achieve maximum benefit from increased query performance, simpler scalability, more flexible management, better availability, and to match the type of store to the data it will hold.</span></span> <span data-ttu-id="e7f11-155">此外，請考慮針對不同類型的資料使用不同類型的資料存放區，並根據針對特定資料類型最佳化的程度，來決定使用的類型。</span><span [資料分割指引](../best-practices/data-partitioning.md)」。</span><span class="sxs-lookup"><span data-stu-id="e7f11-157">For more information, see [Data partitioning
