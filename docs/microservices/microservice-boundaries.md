<span data-ttu-id="71ebf-109">在[上一章](./domain-analysis.md)中，我們已經定義一組供「無人機遞送」應用程式使用的限界內容。</span><span class="sxs-lookup"><span data-stu-id="71ebf-109">In the [previous chapter](./domain-analysis.md), we defined a set of bounded contexts for the Drone Delivery application.</span></span> <span data-ttu-id="71ebf-110">接著，我們更仔細地查看了這些限界內容的其中一個 (也就是「出貨」限界內容)，並找出該限界內容的一組實體、彙總和領域服務。</span><span class="sxs-lookup"><span data-stu-id="71ebf-110">Then we looked more closely at one of these bounded contexts, the Shipping bounded context, and identified a set of entities, aggregates, and domain services for that bounded context.</span></span>

<span data-ttu-id="71ebf-111">而現在，我們已準備好從領域模型階段進入應用程式設計階段。</span><span class="sxs-lookup"><span data-stu-id="71ebf-111">Now we're ready to go from domain model to application design.</span></span> <span data-ttu-id="71ebf-112">以下是可用來從領域模型衍生出微服務的方法。</span><span class="sxs-lookup"><span data-stu-id="71ebf-112">Here's an approach that you can use to derive microservices from the domain model.</span></span>

1. <span data-ttu-id="71ebf-113">從限界內容來開始。</span><span class="sxs-lookup"><span data-stu-id="71ebf-113">Start with a bounded context.</span></span> <span data-ttu-id="71ebf-114">一般來說，微服務中的功能應該不會跨越多個限界內容。</span><span class="sxs-lookup"><span data-stu-id="71ebf-114">In general, the functionality in a microservice should not span more than one bounded context.</span></span> <span data-ttu-id="71ebf-115">根據定義，限界內容會標示特定領域模型的界限。</span><span class="sxs-lookup"><span data-stu-id="71ebf-115">By definition, a bounded context marks the boundary of a particular domain model.</span></span> <span data-ttu-id="71ebf-116">如果您發現某個微服務與不同的領域模型混合在一起，這表示您可能需要回頭重新琢磨您的領域分析。</span><span class="sxs-lookup"><span data-stu-id="71ebf-116">If you find that a microservice mixes different domain models together, that's a sign that you may need to go back and refine your domain analysis.</span></span>

2. <span data-ttu-id="71ebf-117">接著，查看領域模型中的彙總。</span><span class="sxs-lookup"><span data-stu-id="71ebf-117">Next, look at the aggregates in your domain model.</span></span> <span data-ttu-id="71ebf-118">彙總通常很適合作為微服務。</span><span class="sxs-lookup"><span data-stu-id="71ebf-118">Aggregates are often good candidates for microservices.</span></span> <span data-ttu-id="71ebf-119">設計良好的彙總會展現出擁有良好設計之微服務的許多特性，例如：</span><span class="sxs-lookup"><span data-stu-id="71ebf-119">A well-designed aggregate exhibits many of the characteristics of a well-designed microservice, such as:</span></span>

    - <span data-ttu-id="71ebf-120">彙總是衍生自商務需求而非出於技術方面的顧慮 (例如資料存取或傳訊)。</span><span class="sxs-lookup"><span data-stu-id="71ebf-120">An aggregate is derived from business requirements, rather than technical concerns such as data access or messaging.</span></span>
    - <span data-ttu-id="71ebf-121">彙總應該要高度銜接各項功能。</span><span class="sxs-lookup"><span data-stu-id="71ebf-121">An aggregate should have high functional cohesion.</span></span>
    - <span data-ttu-id="71ebf-122">彙總是持續性的界限。</span><span class="sxs-lookup"><span data-stu-id="71ebf-122">An aggregate is a boundary of persistence.</span></span>
    - <span data-ttu-id="71ebf-123">彙總應該要鬆散結合。</span><span class="sxs-lookup"><span data-stu-id="71ebf-123">Aggregates should be loosely coupled.</span></span>

4. <span data-ttu-id="71ebf-128">最後要考慮非功能性的需求。</span><span class="sxs-lookup"><span data-stu-id="71ebf-128">Finally, consider non-functional requirements.</span></span> <span data-ttu-id="71ebf-129">審視各項因素，例如小組規模、資料類型、技術、延展性需求、可用性需求和安全性需求。</span><span class="sxs-lookup"><span data-stu-id="71ebf-129">Look at factors such as team size, data types, technologies, scalability requirements, availability requirements, and security requirements.</span></span> <span data-ttu-id="71ebf-130">這些因素可能會讓您進一步將微服務分解成兩個以上較小的服務，或反過來將數個微服務合併成一個。</span><span class="sxs-lookup"><span data-stu-id="71ebf-130">These factors may lead you to further decompose a microservice into two or more smaller services, or do the opposite and combine several microservices into one.</span></span>

<span data-ttu-id="71ebf-131">在識別應用程式中的各個微服務後，請根據下列準則驗證您的設計：</span><span class="sxs-lookup"><span data-stu-id="71ebf-131">After you identify the microservices in your application, validate your design against the following criteria:</span></span>

- <span data-ttu-id="71ebf-132">每一項服務都各肩負一項責任。</span><span class="sxs-lookup"><span data-stu-id="71ebf-132">Each service has a single responsibility.</span></span>
- <span data-ttu-id="71ebf-133">服務之間不會頻繁地通訊。</span><span class="sxs-lookup"><span data-stu-id="71ebf-133">There are no chatty calls between services.</span></span> <span data-ttu-id="71ebf-134">如果將功能分割成兩個服務會使這兩個服務頻繁通訊，便可能表示這些功能應該放在同一個服務內。</span><span class="sxs-lookup"><span data-stu-id="71ebf-134">If splitting functionality into two services causes them to be overly chatty, it may be a symptom that these functions belong in the same service.</span></span>
- <span data-ttu-id="71ebf-135">每個服務都夠小，以便可由小型小組獨立工作來加以建置。</span><span class="sxs-lookup"><span data-stu-id="71ebf-135">Each service is small enough that it can be built by a small team working independently.</span></span>
- <span data-ttu-id="71ebf-136">沒有交互相依性，因此不必將兩個以上的服務一起部署。</span><span class="sxs-lookup"><span data-stu-id="71ebf-136">There are no inter-dependencies that will require two or more services to be deployed in lock-step.</span></span> <span data-ttu-id="71ebf-137">在部署某項服務時，應該要永遠不必重新部署任何其他服務。</span><span class="sxs-lookup"><span data-stu-id="71ebf-137">It should always be possible to deploy a service without redeploying any other services.</span></span>
- <span data-ttu-id="71ebf-138">服務未緊密結合，並且可以獨立進化。</span><span class="sxs-lookup"><span data-stu-id="71ebf-138">Services are not tightly coupled, and can evolve independently.</span></span>
- <span data-ttu-id="71ebf-139">您的服務界限不會產生資料一致性或完整性的問題。</span><span class="sxs-lookup"><span data-stu-id="71ebf-139">Your service boundaries will not create problems with data consistency or integrity.</span></span> <span data-ttu-id="71ebf-140">有時候您必須透過將功能放入單一微服務中，以保持資料一致性。</span><span class="sxs-lookup"><span data-stu-id="71ebf-140">Sometimes it's important to maintain data consistency by putting functionality into a single microservice.</span></span> <span data-ttu-id="71ebf-141">話雖如此，請考慮您是否真的需要強式一致性。</span><span class="sxs-lookup"><span data-stu-id="71ebf-141">That said, consider whether you really need strong consistency.</span></span> <span data-ttu-id="71ebf-142">您可以找到策略來解決分散式系統的最終一致性問題，而且拆解服務所能帶來的好處，通常會勝過管理最終一致性所帶來的麻煩。</span><span class="sxs-lookup"><span data-stu-id="71ebf-142">There are strategies for addressing eventual consistency in a distributed system, and the benefits of decomposing services often outweigh the challenges of managing eventual consistency.</span></span>

<span data-ttu-id="71ebf-147">請回想一下，開發小組已經識別出四個彙總 (分別是「遞送」、「包裹」、「無人機」和「帳戶」) 以及兩個領域服務 (分別是「排程器」和「監督員」)。</span><span class="sxs-lookup"><span data-stu-id="71ebf-147">Recall that the development team had identified the four aggregates &mdash; Delivery, Package, Drone, and Account &mdash; and two domain services, Scheduler and Supervisor.</span></span>

## <a name="choosing-a-compute-option"></a><span data-ttu-id="71ebf-175">選擇計算選項</span><span class="sxs-lookup"><span data-stu-id="71ebf-175">Choosing a compute option</span></span>

<span data-ttu-id="71ebf-176">*計算*一詞是指您的應用程式執行所在運算資源的裝載模型。</span><span class="sxs-lookup"><span data-stu-id="71ebf-176">The term *compute* refers to the hosting model for the computing resources that your application runs on.</span></span> <span data-ttu-id="71ebf-177">應用在微服務架構方面，有兩種非常熱門的方法：</span><span class="sxs-lookup"><span data-stu-id="71ebf-177">For a microservices architecture, two approaches are especially popular:</span></span>

- <span data-ttu-id="71ebf-178">服務協調器，可管理專用節點 (VM) 上所執行的服務。</span><span class="sxs-lookup"><span data-stu-id="71ebf-178">A service orchestrator that manages services running on dedicated nodes (VMs).</span></span>
