# <a name="enterprise-cloud-adoption-rationalize-the-digital-estate"></a>企業雲端採用：合理化數位資產

雲端合理化是評估資產以判斷何種方法最適合用來將其裝載於雲端的程序。 在確認[方法](approach.md)並彙總[清查](inventory.md)後，即可開始進行雲端合理化。 [合理化的 5R 策略](5-rs-of-rationalization.md)將討論最常用的合理化選項。

## <a name="traditional-view-of-rationalization"></a>合理化的傳統檢視

將傳統的合理化程序視覺化 (例如複雜決策樹) 後，合理化將很容易了解。 數位資產中的每個資產都會經由一項程序傳入，而產生五項解答 (5R) 的其中之一。 此程序非常適用於小型資產。 對於較大的資產則不是很有效率，並且可能導致明顯的延遲。 我們將檢查該程序以了解原因。 然後，我們會產生更有效率的模型。

**清查**。 要使用傳統模型完成完整的合理化，必須徹底清查各項資產，包括應用程式、軟體、硬體、作業系統、系統效能計量等。

**量化分析**。 在決策樹中，量化問題會完成第一層決策。 常見問題：資產目前是否使用中？ 若是如此，它是否已最佳化並適當調整大小？ 資產之間有哪些相依性？ 這些問題都是徹底的清查所不可或缺的。

**定性分析**。 下一組決策須倚賴人類智慧以定性分析的形式產生。 這些問題通常會隨著解決方案而不同，而且只有商務專案關係人和進階使用者才能回答。 這些決策常會使整體進度延遲，導致速度大幅下降。 每個應用程式的這項分析通常需耗時 40&ndash;80 FTE 小時。 如需建置定性分析問題的指引，請參閱[數位資產規劃方法](approach.md)。

**合理化決策**。 如果合理化小組的經驗豐富，定性和量化資料將可建立明確的決策。 只可惜，合理化經驗豐富的小組僱用成本高昂，或是需要好幾個月的事前訓練。

## <a name="rationalization-at-enterprise-scale"></a>企業規模的合理化

如果對 50 個 VM 的數位資產執行這項工作就已十分耗時而令人怯步，那麼在具有數千個 VM 及數百個應用程式的環境中推動商務轉型所需的工作，就令人難以想像了。 所需的人力工時將動輒超過 1,500 FTE 小時，且需要 9 個月的規劃。

雖然完整的合理化是最終狀態，並且可作為後續走向的絕佳指引，但實際上卻鮮少產生與所需的時間和心力成比例的高 ROI。

若合理化對財務決策而言詞十分重要，就值得考慮請專精於雲端合理化的專業服務組織來加速此程序。 即使如此，完整的合理化仍可能是昂貴又耗時的工作，可能會延遲商務轉型或成果。

本文的其餘部分將說明其替代方法，一般稱之為漸進式合理化。

## <a name="incremental-rationalization"></a>漸進式合理化

大型數位資產的完整合理化很容易發生風險，並可能因複雜度而導致延遲的狀況。 漸進式方法的基礎假設是，延遲的決策可錯開企業的負擔，而降低發生障礙的風險。 經過一段時間後，此方法會形成有機模組，用以開發擬定有效合理化決策並提升其執行效率所需的程序和體驗。

### <a name="inventory-reduce-discovery-data-points"></a>清查：減少探索資料點

很少有組織會投入時間、心力和費用來維護精確而即時的完整數位資產清查。 舉凡遺失、遭竊、更新週期，和新進員工到職，都可說是使用者裝置需進行詳細資產追蹤的原因。 不過，在傳統的內部部署資料中心維護精確的伺服器和應用程式清查，投資報酬率 (ROI) 通常很低。 相較於在資料中心追蹤固定資產的使用情形，大部分的 IT 組織都有其他更迫切的問題需要處理。

在進行雲端轉換時，清查會直接與營運成本相互關聯。 必須要有精確的清查資料，才能完成妥善規劃。 但很可惜，目前的環境掃描選項為了掃描完整清查並加編目，可能會使決策延遲達週數或數個月。 幸而，我們可以透過一些技巧來加速資料收集。

代理程式型掃描是最常被提到的延遲。 傳統合理化所需的完整資料，通常依存於可經由在每個資產上執行的代理程式來收集的資料。 這種對於代理程式的相依性常會拖慢進度，因為它可能需要來自於安全性、作業及管理功能的反饋。

在漸進式合理化程序中，可使用無代理程式解決方案來進行初始探索，以加速完成早期的決策。 根據環境中的複雜程度，仍有可能需要使用代理程式型解決方案，但已非商務變更的關鍵環節。

### <a name="quantitative-analysis-streamline-decisions"></a>量化分析：簡化決策

無論採用何種方法進行清查探索，量化分析都可產生多項初始決策和假設。 尤其是在嘗試找出第一個工作負載時，或合理化的目標是高階成本比較時，更是如此。 在漸進式合理化的程序中，「雲端策略」和「雲端採用」小組會嚴格依據[合理化的 5R 策略](5-rs-of-rationalization.md)產生兩項簡要的決策，並僅套用這些量化因素，進而簡化分析並減少推動變更所需的初始資料量。

例如，如果組織正在將 IaaS 移轉至雲端，則可以假設大部分的工作負載都將汰除或重新裝載。

### <a name="qualitative-analysis-temporary-assumptions"></a>定性分析：暫時性假設

藉由減少潛在結果數量，與資產的未來狀態有關的初始決策將更容易達成。 當選項變少時，在這個早期階段對企業提出的問題數也會隨之減少。

繼續使用前例，如果選項僅限於重新裝載或淘汰，則在初始合理化期間只需詢問企業一個問題 &mdash; 即是否要淘汰。

「分析指出目前沒有使用者主動使用此資產。 這是正確的資訊嗎？還是我們疏漏了什麼？」 一般而言，用這種二元問題來執行定性分析會簡單許多。

這個精簡方法會產生基準、財務計劃、策略和方向。 在後續的活動中，每項資產都會執行進一步的合理化和定性分析，以評估其他選項。 在此初始合理化中所做的所有假設都會先經過測試，再進行實作。

## <a name="challenging-assumptions"></a>挑戰假設

上一節產生的結果是代入假設而執行的粗略合理化。 接下來，我們可以挑戰其中的某些假設。

### <a name="retiring-assets"></a>淘汰資產

在傳統的內部部署環境中，裝載小型、未使用的資產鮮少會對年度成本造成重大影響。 除了少數例外，因剪除和淘汰這些資產而達成的成本樽節，遠不及為了分析及淘汰實際資產所需執行的 FTE 工作。

不過，在移至雲端帳戶處理模型後，淘汰資產將可省下可觀的年度營運成本和前置移轉工作。

在完成量化分析後淘汰 20% 或更多數位資產的組織不在少數。 在決定執行這類動作之前，建議先完成進一步的定性分析。 一經確認後，淘汰這些資產即可能產生完成雲端移轉後的投資報酬首勝。 這經常都是最重要的成本節省因素之一。 因此我們建議，在移轉程序的建置階段執行期間，「雲端策略」小組應同時監看資產的驗證和淘汰程序，以獲取初期的財務勝績。

### <a name="program-adjustments"></a>方案調整

很少有公司只進行一個轉型程序。 在降低成本、追求市場成長與新增收益來源之間所做的抉擇，鮮少是二元決策。 因此，建議雲端策略小組在主要轉型程序的範圍外執行轉型工作的同時，應與 IT 人員合作進行資產識別。

在本文所使用的 IaaS 移轉範例中：

- 要求 DevOps 小組識別已納入部署自動化程序中的資產，並從核心移轉計劃中移除這些資產。

- 要求資料和研發小組識別帶動新收益來源的資產，並將其從核心移轉計劃中移除。

這種以方案為焦點的定性分析可快速執行，並且可讓多個移轉待處理項目達成一致性。

在初始移轉之後，某些資產可能仍需要被視為重新裝載資產一段時間，以分階段導入後續的合理化。

## <a name="selecting-the-first-workload"></a>選取第一個工作負載

實作第一個工作負載是測試和學習的關鍵。 這是展現及建立成長型思維的首次機會。

### <a name="business-criteria"></a>商務準則

識別由雲端策略小組的業務單位成員所支援的工作負載，以確保業務透明性。 最好是選擇其所屬小組對於移至雲端有多方面參與和強烈動機的工作負載。

### <a name="technical-criteria"></a>技術準則

選取相依性最低、且能夠以小型資產群組的形式移動的工作負載。 建議您選取已定義測試路徑的工作負載，以簡化驗證程序。

第一個工作負載通常會部署在實驗環境中，且不具操作或管理能力。 請務必選取不會與安全資料互動的工作負載。

### <a name="qualitative-analysis"></a>定性分析

「雲端採用」和「雲端策略」小組可互相合作，共同分析這個小型工作負載。 這會產生一個建立和測試定性分析準則的受控機會。 較小的母體會產生對受影響的使用者進行調查的機會，以在一週內完成詳細的定性分析。 若想了解共通的定性分析因素，請參閱[合理化的 5R 策略](5-rs-of-rationalization.md)中的特定合理化目標。

### <a name="migration"></a>移轉

在持續進行合理化的同時，「雲端採用」小組也開始移轉小型工作負載，以擴充下列主要區域的學習：

- 透過雲端提供者的平台強化技能。
- 定義達成長期願景所需的核心服務 (和 Azure 標準)。
- 深入了解營運在後續的轉型過程中可能需要怎樣的發展。
- 了解任何既有的商業風險，以及企業對這些風險的承受能力。
- 根據企業對風險的承受能力，建立控管的基準或最小可行性產品 (MVP)

## <a name="release-planning"></a>版本規劃

當「雲端採用」小組執行移轉或實作第一個工作負載時，「雲端策略」小組即可開始設定其餘應用程式/工作負載的優先順序。

### <a name="power-of-ten"></a>以十為單位

傳統方法會窮盡一切力量進行合理化。 所幸，要開始進行一個轉型程序，通常不需要為每個應用程式設定一個計劃。 在漸進式模型中，「以十為單位」可作為理想的起始點。 在此模型中，雲端策略小組會選取要移轉的前十個應用程式。 這十個工作負載應會同時包含簡單和複雜的工作負載。

### <a name="building-the-first-backlogs"></a>建置第一個待處理項目

「雲端採用」和「雲端策略」小組可互相合作，共同進行前十個工作負載的定性分析。 這會產生第一優先的移轉待處理項目，和第一優先的發行待處理項目。 這個方法可讓各小組反覆執行此方法，並提供足夠的時間來建立定性分析的適當程序。

### <a name="maturing-the-process"></a>發展穩定程序

在兩個小組皆認可定性分析準則後，評量即可成為每次反覆執行時的的工作之一。 要達成評量準則的共識，通常需要 2&ndash;3 次的發行。

在評量進入移轉的漸進式執行程序之後，「雲端採用」小組即可更快速地反覆執行評量和架構。 在這個階段中也會抽離「雲端策略」小組，以避免過度耗損其時間。 這也可讓「雲端策略」小組專注於在特定發行中尚未完成的應用程式優先順序設定，以確保能密切因應持續變化的市場狀況。

並非所有已排定優先順序的應用程式都已可進行移轉。 隨著小組執行更深入的定性分析，並發現可能需要您重新排定待辦項目優先順序的商務事件和相依性時，排序有可能會變更。 某些發行可能會將少量的工作負載分組在一起。 有些則可能只包含單一工作負載。

「雲端採用」小組所執行的反覆運作可能不會產生完成的工作負載移轉。 工作負載越小，相依性就越少，工作負載可納入單一短期衝刺或反覆執行中的可能性就越高。 基於此原因，建議您在發行待處理項目中的前幾個應用程式應該要較小，且僅包含少量外部相依性。

## <a name="end-state"></a>最終狀態

經過一段時間後，「雲端採用」和「雲端策略」小組將共同完成清查的完整合理化。 不過，這種漸進式方法可讓小組持續加快合理化程序的執行速度。 它也可讓轉型程序更快產生實際的商務成果，而不需要像前置分析工作一樣那麼費事。

在某些情況下，財務模型的資料可能會太過密集，若沒有額外的合理化，就無法制定可實行的決策。 在這種情況下，可能需要以較傳統的方法進行合理化。
