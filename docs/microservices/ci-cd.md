# <a name="designing-microservices-continuous-integration"></a><span data-ttu-id="333d2-103">設計微服務：持續整合</span><span class="sxs-lookup"><span data-stu-id="333d2-103">Designing microservices: Continuous integration</span></span>

<span data-ttu-id="333d2-104">持續整合與持續傳遞 (CI/CD) 是透過微服務獲得成功的關鍵需求。</span><span class="sxs-lookup"><span data-stu-id="333d2-104">Continuous integration and continuous delivery (CI/CD) are a key requirement for achieving success with microservices.</span></span> <span data-ttu-id="333d2-105">若沒有完善的 CI/CD 程序，您就無法達到微服務所承諾的靈活度。</span><span class="sxs-lookup"><span data-stu-id="333d2-105">Without a good CI/CD process, you will not achieve the agility that microservices promise.</span></span> <span data-ttu-id="333d2-106">微服務的某些 CI/CD 挑戰是來自於各種服務的多個程式碼基底及各式各樣的建置環境。</span><span class="sxs-lookup"><span data-stu-id="333d2-106">Some of the CI/CD challenges for microservices arise from having multiple code bases and heterogenous build environments for the various services.</span></span> <span data-ttu-id="333d2-107">本章描述一些挑戰，並建議可解決問題的一些方法。</span><span class="sxs-lookup"><span data-stu-id="333d2-107">This chapter describes the challenges and recommends some approaches to the problem.</span></span>

![](./images/ci-cd.png)

<span data-ttu-id="333d2-108">更快速的發行週期是採用微服務架構的最大原因之一。</span><span class="sxs-lookup"><span data-stu-id="333d2-108">Faster release cycles are one of the biggest reasons to adopt a microservices architecture.</span></span> 

<span data-ttu-id="333d2-109">純粹單體式應用程式中有一個建置管線，其輸出是應用程式可執行檔。</span><span class="sxs-lookup"><span data-stu-id="333d2-109">In a purely monolithic application, there is a single build pipeline whose output is the application executable.</span></span> <span data-ttu-id="333d2-110">所有開發工作都會饋送到此管線中。</span><span class="sxs-lookup"><span data-stu-id="333d2-110">All development work feeds into this pipeline.</span></span> <span data-ttu-id="333d2-111">如果發現高優先順序的錯誤 (bug)，則必須整合、測試及發行修正程式，這可能會延遲新功能的發行。</span><span class="sxs-lookup"><span data-stu-id="333d2-111">If a high-priority bug is found, a fix must be integrated, tested, and published, which can delay the release of new features.</span></span> <span data-ttu-id="333d2-112">您的確可藉由擁有構造良好的模組以及使用功能分支來緩和這些問題，進而將程式碼變更的影響降到最低。</span><span class="sxs-lookup"><span data-stu-id="333d2-112">It's true that you can mitigate these problems by having well-factored modules and using feature branches to minimize the impact of code changes.</span></span> <span data-ttu-id="333d2-113">但隨著應用程式日益複雜，並新增了更多功能，單體的發行程序就變得更加脆弱並可能中斷。</span><span class="sxs-lookup"><span data-stu-id="333d2-113">But as the application grows more complex, and more features are added, the release process for a monolith tends to become more brittle and likely to break.</span></span> 

<span data-ttu-id="333d2-114">根據微服務的原理，絕不會有每個小組都必須排隊的冗長發行序列。</span><span class="sxs-lookup"><span data-stu-id="333d2-114">Following the microservices philosophy, there should never be a long release train where every team has to get in line.</span></span> <span data-ttu-id="333d2-115">建立服務 "A" 的小組可以隨時發行更新，而不需等待服務 "B" 的變更來進行合併、測試和部署。</span><span class="sxs-lookup"><span data-stu-id="333d2-115">The team that builds service "A" can release an update at any time, without waiting for changes in service "B" to be merged, tested, and deployed.</span></span> <span data-ttu-id="333d2-116">CI/CD 程序是達成此目的的關鍵。</span><span class="sxs-lookup"><span data-stu-id="333d2-116">The CI/CD process is critical to making this possible.</span></span> <span data-ttu-id="333d2-117">發行管線必須自動化且高度可靠，才能將部署更新的風險降到最低。</span><span class="sxs-lookup"><span data-stu-id="333d2-117">Your release pipeline must be automated and highly reliable, so that the risks of deploying updates are minimized.</span></span> <span data-ttu-id="333d2-118">如果您是每天或一天多次地對生產環境進行發行，則退回或服務中斷的狀況必定要非常罕見。</span><span class="sxs-lookup"><span data-stu-id="333d2-118">If you are releasing to production daily or multiple times a day, regressions or service disruptions must be very rare.</span></span> <span data-ttu-id="333d2-119">同時，如果部署了錯誤的更新，您必須有可靠的方法能快速復原或向前復原至舊版的服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-119">At the same time, if a bad update does get deployed, you must have a reliable way to quickly roll back or roll forward to a previous version of a service.</span></span>

![](./images/cicd-monolith.png)

<span data-ttu-id="333d2-120">當我們談論 CI/CD 時，我們實際會談論數個相關程序：持續整合、持續傳遞和持續部署。</span><span class="sxs-lookup"><span data-stu-id="333d2-120">When we talk about CI/CD, we are really talking about several related processes: Continuous integration, continuous delivery, and continuous deployment.</span></span>

- <span data-ttu-id="333d2-121">持續整合意味著程式碼變更經常會併入主要分支中，使用自動化建置和測試程序來確保主要分支中的程式碼一律具有良好的生產品質。</span><span class="sxs-lookup"><span data-stu-id="333d2-121">Continuous integration means that code changes are frequently merged into the main branch, using automated build and test processes to ensure that  code in the main branch is always production-quality.</span></span>

- <span data-ttu-id="333d2-122">持續傳遞意味著傳遞 CI 程序的程式碼變更會自動發佈到類似生產的環境。</span><span class="sxs-lookup"><span data-stu-id="333d2-122">Continuous delivery means that code changes that pass the CI process are automatically published to a production-like environment.</span></span> <span data-ttu-id="333d2-123">部署至即時生產環境可能需要手動核准，除此之外皆是自動化作業。</span><span class="sxs-lookup"><span data-stu-id="333d2-123">Deployment into the live production environment may require manual approval, but is otherwise automated.</span></span> <span data-ttu-id="333d2-124">這麼做的目的是，您的程式碼應該總是「準備好」部署到生產環境。</span><span class="sxs-lookup"><span data-stu-id="333d2-124">The goal is that your code should always be *ready* to deploy into production.</span></span>

- <span data-ttu-id="333d2-125">持續部署意味著傳遞 CI/CD 程序的程式碼變更會自動發佈到生產環境中。</span><span class="sxs-lookup"><span data-stu-id="333d2-125">Continuous deployment means that code changes that pass the CI/CD process are automatically deployed into production.</span></span>

<span data-ttu-id="333d2-126">在 Kubernetes 和微服務的內容中，CI 階段涉及建立和測試容器映像，並將這些映像推送到容器登錄。</span><span class="sxs-lookup"><span data-stu-id="333d2-126">In the context of Kubernetes and microservices, the CI stage is concerned with building and testing container images, and pushing those images to a container registry.</span></span> <span data-ttu-id="333d2-127">在部署階段中，pod 規格會進行更新，以取得最新的生產映像。</span><span class="sxs-lookup"><span data-stu-id="333d2-127">In the deployment stage, pod specs are updated to pick up the latest production image.</span></span>

## <a name="challenges"></a><span data-ttu-id="333d2-128">挑戰</span><span class="sxs-lookup"><span data-stu-id="333d2-128">Challenges</span></span>

- <span data-ttu-id="333d2-129">**許多小型獨立的程式碼基底**。</span><span class="sxs-lookup"><span data-stu-id="333d2-129">**Many small independent code bases**.</span></span> <span data-ttu-id="333d2-130">每個小組需利用自己的建置管線，負責建立自己的服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-130">Each team is responsible for building its own service, with its own build pipeline.</span></span> <span data-ttu-id="333d2-131">在某些組織中，小組可以使用不同的程式碼存放庫。</span><span class="sxs-lookup"><span data-stu-id="333d2-131">In some organizations, teams may use separate code repositories.</span></span> <span data-ttu-id="333d2-132">這可能會導致一個情況，就是如何建置系統的知識已散佈到各小組，但組織中沒有人知道如何部署整個應用程式。</span><span class="sxs-lookup"><span data-stu-id="333d2-132">This could lead to a situation where the knowledge of how to build the system is spread across teams, and nobody in the organization knows how to deploy the entire application.</span></span> <span data-ttu-id="333d2-133">例如，如果您需要快速部署到新的叢集，災害復原案例會發生什麼情況？</span><span class="sxs-lookup"><span data-stu-id="333d2-133">For example, what happens in a disaster recovery scenario, if you need to quickly deploy to a new cluster?</span></span>   

- <span data-ttu-id="333d2-134">**多個語言和架構**。</span><span class="sxs-lookup"><span data-stu-id="333d2-134">**Multiple languages and frameworks**.</span></span> <span data-ttu-id="333d2-135">如果每個小組都使用自己的各項技術，則可能很難建立適用於整個組織的單一建置程序。</span><span class="sxs-lookup"><span data-stu-id="333d2-135">With each team using its own mix of technologies, it can be difficult to create a single build process that works across the organization.</span></span> <span data-ttu-id="333d2-136">建置程序必須有足夠的彈性，每個小組才能針對其選擇的語言或架構進行調整。</span><span class="sxs-lookup"><span data-stu-id="333d2-136">The build process must be flexible enough that every team can adapt it for their choice of language or framework.</span></span> 

- <span data-ttu-id="333d2-137">**整合及負載測試**。</span><span class="sxs-lookup"><span data-stu-id="333d2-137">**Integration and load testing**.</span></span> <span data-ttu-id="333d2-138">如果小組依照自己的步調發行更新，則要設計強固的端對端測試可能具有挑戰性，特別是在服務相依於其他服務時。</span><span class="sxs-lookup"><span data-stu-id="333d2-138">With teams releasing updates at their own pace, it can be challenging to design robust end-to-end testing, especially when services have dependencies on other services.</span></span> <span data-ttu-id="333d2-139">此外，執行完整生產叢集可能很耗費資源，因此，如果只作為測試用，不太可能每個小組都能夠以生產規模執行自己的完整叢集。</span><span class="sxs-lookup"><span data-stu-id="333d2-139">Moreover, running a full production cluster can be expensive, so it's unlikely that every team will be able to run its own full cluster at production scales, just for testing.</span></span> 

- <span data-ttu-id="333d2-140">**發行管理**。</span><span class="sxs-lookup"><span data-stu-id="333d2-140">**Release management**.</span></span> <span data-ttu-id="333d2-141">每個小組應該都要能夠將更新部署到生產環境。</span><span class="sxs-lookup"><span data-stu-id="333d2-141">Every team should have the ability to deploy an update to production.</span></span> <span data-ttu-id="333d2-142">但這不表示每個小組成員都有權限這麼做。</span><span class="sxs-lookup"><span data-stu-id="333d2-142">That doesn't mean that every team member has permissions to do so.</span></span> <span data-ttu-id="333d2-143">不過，擁有集中的發行管理員角色可能會降低部署的速度。</span><span class="sxs-lookup"><span data-stu-id="333d2-143">But having a centralized Release Manager role can reduce the velocity of deployments.</span></span> <span data-ttu-id="333d2-144">因此您的 CI/CD 程序愈加自動化和可靠，中央授權的需求就愈低。</span><span class="sxs-lookup"><span data-stu-id="333d2-144">The more that your CI/CD process is automated and reliable, the less there should be a need for a central authority.</span></span> <span data-ttu-id="333d2-145">儘管如此，您可能使用不同的原則來發行主要功能更新和次要錯誤修正。</span><span class="sxs-lookup"><span data-stu-id="333d2-145">That said, you might have different policies for releasing major feature updates versus minor bug fixes.</span></span> <span data-ttu-id="333d2-146">分散管理並不表示就是零治理。</span><span class="sxs-lookup"><span data-stu-id="333d2-146">Being decentralized does not mean there should be zero governance.</span></span>

- <span data-ttu-id="333d2-147">**容器映像版本控制**。</span><span class="sxs-lookup"><span data-stu-id="333d2-147">**Container image versioning**.</span></span> <span data-ttu-id="333d2-148">在開發和測試週期內，CI/CD 程序將會建立許多容器映像。</span><span class="sxs-lookup"><span data-stu-id="333d2-148">During the development and test cycle, the CI/CD process will build many container images.</span></span> <span data-ttu-id="333d2-149">其中只有一些映像是發行候選項目，然而只有其中一些發行候選項目會推送到生產環境中。</span><span class="sxs-lookup"><span data-stu-id="333d2-149">Only some of those are candidates for release, and then only some of those release candidates will get pushed into production.</span></span> <span data-ttu-id="333d2-150">您應該有明確的版本控制策略，好讓您知道哪些映像目前已部署至生產環境，而且可以視需要復原至先前的版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-150">You should have a clear versioning strategy, so that you know which images are currently deployed to production, and can roll back to a previous version if necessary.</span></span> 

- <span data-ttu-id="333d2-151">**服務更新**。</span><span class="sxs-lookup"><span data-stu-id="333d2-151">**Service updates**.</span></span> <span data-ttu-id="333d2-152">當您將服務更新為新版本時，其不得中斷與其相依的其他服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-152">When you update a service to a new version, it shouldn't break other services that depend on it.</span></span> <span data-ttu-id="333d2-153">如果您執行輪流更新，則會有一段時間執行混合的版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-153">If you do a rolling update, there will be a period of time when a mix of versions is running.</span></span> 
 
<span data-ttu-id="333d2-154">這些挑戰反映出主要壓力。</span><span class="sxs-lookup"><span data-stu-id="333d2-154">These challenges reflect a fundamental tension.</span></span> <span data-ttu-id="333d2-155">一方面，小組必須盡可能獨立運作。</span><span class="sxs-lookup"><span data-stu-id="333d2-155">On the one hand, teams need to work as independently as possible.</span></span> <span data-ttu-id="333d2-156">另一方面則需要一些協調，以便單一人員執行一些工作，例如執行整合測試、將整個解決方案重新部署到新叢集，或復原錯誤的更新。</span><span class="sxs-lookup"><span data-stu-id="333d2-156">On the other hand, some coordination is needed so that a single person can do tasks like running an integration test, redeploying the entire solution to a new cluster, or rolling back a bad update.</span></span> 
 
## <a name="cicd-approaches-for-microservices"></a><span data-ttu-id="333d2-157">微服務的 CI/CD 方法</span><span class="sxs-lookup"><span data-stu-id="333d2-157">CI/CD approaches for microservices</span></span>

<span data-ttu-id="333d2-158">每個服務小組最好能將其建置環境容器化。</span><span class="sxs-lookup"><span data-stu-id="333d2-158">It's a good practice for every service team to containerize their build environment.</span></span> <span data-ttu-id="333d2-159">此容器應具有為其服務建立程式碼成品所需的所有建置工具。</span><span class="sxs-lookup"><span data-stu-id="333d2-159">This container should have all of the build tools necessary to build the code artifacts for their service.</span></span> <span data-ttu-id="333d2-160">您通常可以找到您語言和架構的正式 Docker 映像。</span><span class="sxs-lookup"><span data-stu-id="333d2-160">Often you can find an official Docker image for your language and framework.</span></span> <span data-ttu-id="333d2-161">然後您可以使用 `docker run` 或 Docker Compose 來執行組建。</span><span class="sxs-lookup"><span data-stu-id="333d2-161">Then you can use `docker run` or Docker Compose to run the build.</span></span> 

<span data-ttu-id="333d2-162">使用此方法時，設定新建置環境是件小事。</span><span class="sxs-lookup"><span data-stu-id="333d2-162">With this approach, it's trivial to set up a new build environment.</span></span> <span data-ttu-id="333d2-163">為您建立程式碼的開發人員並不需要安裝一組建置工具，只要執行容器映像即可。</span><span class="sxs-lookup"><span data-stu-id="333d2-163">A developer who wants to build your code doesn't need to install a set of build tools, but simply runs the container image.</span></span> <span data-ttu-id="333d2-164">可能更重要的是，您的組建伺服器可設定為執行相同的動作。</span><span class="sxs-lookup"><span data-stu-id="333d2-164">Perhaps more importantly, your build server can be configured to do the same thing.</span></span> <span data-ttu-id="333d2-165">這樣一來，您就不需要將這些工具安裝到組建伺服器，或管理衝突的工具版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-165">That way, you don't need to install those tools onto the build server, or manage conflicting versions of tools.</span></span> 

<span data-ttu-id="333d2-166">進行本機開發和測試時，可使用 Docker 在容器內執行服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-166">For local development and testing, use Docker to run the service inside a container.</span></span> <span data-ttu-id="333d2-167">在此程序中，您可能需要執行其他容器，因為這些容器具有本機測試所需的模擬服務或測試資料庫。</span><span class="sxs-lookup"><span data-stu-id="333d2-167">As part of this process, you may need to run other containers that have mock services or test databases needed for local testing.</span></span> <span data-ttu-id="333d2-168">您可以使用 Docker Compose 來協調這些容器，或使用 Minikube 在本機執行 Kubernetes。</span><span class="sxs-lookup"><span data-stu-id="333d2-168">You could use Docker Compose to coordinate these containers, or use Minikube to run Kubernetes locally.</span></span> 

<span data-ttu-id="333d2-169">備妥程式碼時，發起提取要求並且併入主檔中。</span><span class="sxs-lookup"><span data-stu-id="333d2-169">When the code is ready, open a pull request and merge into master.</span></span> <span data-ttu-id="333d2-170">這會在組建伺服器上啟動一項作業：</span><span class="sxs-lookup"><span data-stu-id="333d2-170">This will start a job on the build server:</span></span>

2. <span data-ttu-id="333d2-172">對程式碼執行單元測試。</span><span class="sxs-lookup"><span data-stu-id="333d2-172">Run unit tests against the code.</span></span>
3. <span data-ttu-id="333d2-173">建置容器映像。</span><span class="sxs-lookup"><span data-stu-id="333d2-173">Build the container image.</span></span>

6. <span data-ttu-id="333d2-177">使用新的映像來更新測試叢集，以執行整合測試。</span><span class="sxs-lookup"><span data-stu-id="333d2-177">Update the test cluster with the new image to run integration tests.</span></span>

<span data-ttu-id="333d2-178">當映像準備好移入生產環境時，請視需要更新部署檔案，以指定最新的映像，包括任何 Kubernetes 設定檔。</span><span class="sxs-lookup"><span data-stu-id="333d2-178">When the image is ready to go into production, update the deployment files as needed to specify the latest image, including any Kubernetes configuration files.</span></span> <span data-ttu-id="333d2-179">然後將更新套用到生產叢集。</span><span class="sxs-lookup"><span data-stu-id="333d2-179">Then apply the update to the production cluster.</span></span>

<span data-ttu-id="333d2-180">以下是讓部署更可靠的一些建議：</span><span class="sxs-lookup"><span data-stu-id="333d2-180">Here are some recommendations for making deployments more reliable:</span></span>

## <a name="updating-services"></a><span data-ttu-id="333d2-186">更新服務</span><span class="sxs-lookup"><span data-stu-id="333d2-186">Updating services</span></span>

<span data-ttu-id="333d2-187">有各種策略可用來更新已在生產環境中的服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-187">There are various strategies for updating a service that's already in production.</span></span> <span data-ttu-id="333d2-188">我們在此討論三個常見的選項：輪流更新、藍綠色部署和 Canary 版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-188">Here we discuss three common options: Rolling update, blue-green deployment, and canary release.</span></span>

### <a name="rolling-update"></a><span data-ttu-id="333d2-189">輪流更新</span><span class="sxs-lookup"><span data-stu-id="333d2-189">Rolling update</span></span> 

<span data-ttu-id="333d2-190">在輪流更新中，您會部署新的服務執行個體，而新的執行個體會立刻開始接收要求。</span><span class="sxs-lookup"><span data-stu-id="333d2-190">In a rolling update, you deploy new instances of a service, and the new instances start receiving requests right away.</span></span> <span data-ttu-id="333d2-191">隨著新執行個體的出現，系統會移除先前的執行個體。</span><span class="sxs-lookup"><span data-stu-id="333d2-191">As the new instances come up, the previous instances are removed.</span></span>

<span data-ttu-id="333d2-192">當您更新部署的 pod 規格時，輪流更新是 Kubernetes 中的預設行為。</span><span class="sxs-lookup"><span data-stu-id="333d2-192">Rolling updates are the default behavior in Kubernetes when you update the pod spec for a Deployment.</span></span> <span data-ttu-id="333d2-193">部署控制器會為已更新的 pod 建立新的 ReplicaSet。</span><span class="sxs-lookup"><span data-stu-id="333d2-193">The Deployment controller creates a new ReplicaSet for the updated pods.</span></span> <span data-ttu-id="333d2-194">然後它會相應增加新的 ReplicaSet，同時相應減少舊的 ReplicaSet，以維持所需的複本計數。</span><span class="sxs-lookup"><span data-stu-id="333d2-194">Then it scales up the new ReplicaSet while scaling down the old one, to maintain the desired replica count.</span></span> <span data-ttu-id="333d2-195">在新的 pod 備妥之前，舊的 pod 不會遭到刪除。</span><span class="sxs-lookup"><span data-stu-id="333d2-195">It doesn't delete old pods until the new ones are ready.</span></span> <span data-ttu-id="333d2-196">Kubernetes 會保存更新歷程記錄，因此您可以視需要使用 kubectl 來復原更新。</span><span class="sxs-lookup"><span data-stu-id="333d2-196">Kubernetes keeps a history of the update, so you can use kubectl to roll back an update if needed.</span></span> 

<span data-ttu-id="333d2-197">如果您的服務執行長時間的啟動工作，您可以定義整備探查。</span><span class="sxs-lookup"><span data-stu-id="333d2-197">If your service performs a long startup task, you can define a readiness probe.</span></span> <span data-ttu-id="333d2-198">當容器準備好開始接收流量時，就會回報整備探查。</span><span class="sxs-lookup"><span data-stu-id="333d2-198">The readiness probe reports when the container is ready to start receiving traffic.</span></span> <span data-ttu-id="333d2-199">在探查回報成功之前，Kubernetes 不會將流量傳送至 pod。</span><span class="sxs-lookup"><span data-stu-id="333d2-199">Kubernetes won't send traffic to the pod until the probe reports success.</span></span> 

<span data-ttu-id="333d2-200">輪流更新的其中一項挑戰就是在更新過程中，新舊版本會混合執行及接收流量。</span><span class="sxs-lookup"><span data-stu-id="333d2-200">One challenge of rolling updates is that during the update process, a mix of old and new versions are running and receiving traffic.</span></span> <span data-ttu-id="333d2-201">在這段期間，所有要求會路由傳送至任一個版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-201">During this period, any request could get routed to either of the two versions.</span></span> <span data-ttu-id="333d2-202">根據兩個版本之間的變動範圍而定，這或許可能會造成問題。</span><span class="sxs-lookup"><span data-stu-id="333d2-202">That may or may not cause problems, depending on the scope of the changes between the two versions.</span></span> 

### <a name="blue-green-deployment"></a><span data-ttu-id="333d2-203">藍綠部署</span><span class="sxs-lookup"><span data-stu-id="333d2-203">Blue-green deployment</span></span>

<span data-ttu-id="333d2-204">在藍綠色部署中，您可以一起部署新版本與前一個版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-204">In a blue-green deployment, you deploy the new version alongside the previous version.</span></span> <span data-ttu-id="333d2-205">在您驗證新版本之後，立刻將所有流量從前一個版本切換到新版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-205">After you validate the new version, you switch all traffic at once from the previous version to the new version.</span></span> <span data-ttu-id="333d2-206">切換之後，您可監視應用程式是否有任何問題。</span><span class="sxs-lookup"><span data-stu-id="333d2-206">After the switch, you monitor the application for any problems.</span></span> <span data-ttu-id="333d2-207">如果發生錯誤，您可以換回舊版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-207">If something goes wrong, you can swap back to the old version.</span></span> <span data-ttu-id="333d2-208">假設沒有任何問題，您可以刪除舊版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-208">Assuming there are no problems, you can delete the old version.</span></span>

<span data-ttu-id="333d2-209">使用更傳統的單體式或多層式架構應用程式，藍綠部署通常是指佈建兩個相同的環境。</span><span class="sxs-lookup"><span data-stu-id="333d2-209">With a more traditional monolithic or N-tier application, blue-green deployment generally meant provisioning two identical environments.</span></span> <span data-ttu-id="333d2-210">您會將新版本部署到預備環境，然後將用戶端流量重新導向至預備環境 &mdash; 例如，藉由交換 VIP 位址。</span><span class="sxs-lookup"><span data-stu-id="333d2-210">You would deploy the new version to a staging environment, then redirect client traffic to the staging environment &mdash; for example, by swapping VIP addresses.</span></span>

<span data-ttu-id="333d2-211">在 Kubernetes 中，您不需佈建個別的叢集來進行藍綠部署。</span><span class="sxs-lookup"><span data-stu-id="333d2-211">In Kubernetes, you don't need to provision a separate cluster to do blue-green deployments.</span></span> <span data-ttu-id="333d2-212">您反而可以利用選取器。</span><span class="sxs-lookup"><span data-stu-id="333d2-212">Instead, you can take advantage of selectors.</span></span> <span data-ttu-id="333d2-213">利用新的 pod 規格和一組不同的標籤，建立新的部署資源。</span><span class="sxs-lookup"><span data-stu-id="333d2-213">Create a new Deployment resource with a new pod spec and a different set of labels.</span></span> <span data-ttu-id="333d2-214">建立此部署，而不需刪除先前的部署，或修改指向它的服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-214">Create this deployment, without deleting the previous deployment or modifying the service that points to it.</span></span> <span data-ttu-id="333d2-215">當新的 pod 執行時，您可以更新服務的選取器，以符合新的部署。</span><span class="sxs-lookup"><span data-stu-id="333d2-215">Once the new pods are running, you can update the service's selector to match the new deployment.</span></span> 

<span data-ttu-id="333d2-216">藍綠部署的優點是服務可同時切換所有 pod。</span><span class="sxs-lookup"><span data-stu-id="333d2-216">An advantage of blue-green deployments is that the service switches all the pods at the same time.</span></span> <span data-ttu-id="333d2-217">更新此服務後，所有新的要求都會路由傳送至新的版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-217">After the service is updated, all new requests get routed to the new version.</span></span> <span data-ttu-id="333d2-218">但有一個缺點，在更新期間您會對服務執行兩倍的 pod 數目 (目前和下一個)。</span><span class="sxs-lookup"><span data-stu-id="333d2-218">One drawback is that during the update, you are running twice as many pods for the service (current and next).</span></span> <span data-ttu-id="333d2-219">如果 pod 需要大量 CPU 或記憶體資源，您可能需要暫時相應放大叢集，以處理資源耗用。</span><span class="sxs-lookup"><span data-stu-id="333d2-219">If the pods require a lot of CPU or memory resources, you may need to scale out the cluster temporarily to handle the resource consumption.</span></span> 

### <a name="canary-release"></a><span data-ttu-id="333d2-220">Canary 版本</span><span class="sxs-lookup"><span data-stu-id="333d2-220">Canary release</span></span>

<span data-ttu-id="333d2-221">在 Canary 版本中，您會向少數用戶端推出更新的版本。</span><span class="sxs-lookup"><span data-stu-id="333d2-221">In a canary release, you roll out an updated version to a small number of clients.</span></span> <span data-ttu-id="333d2-222">然後您會先監視新服務的行為，然後再向所有的用戶端推出該服務。</span><span class="sxs-lookup"><span data-stu-id="333d2-222">Then you monitor the behavior of the new service before rolling it out to all clients.</span></span> <span data-ttu-id="333d2-223">這可讓您以控制方式進行緩慢推出、觀察實際資料，以及在所有客戶受影響之前找出問題。</span><span class="sxs-lookup"><span data-stu-id="333d2-223">This lets you do a slow rollout in a controlled fashion, observe real data, and spot problems before all customers are affected.</span></span>
