# <a name="designing-microservices-domain-analysis"></a><span data-ttu-id="cc8c1-103">設計微服務：網域分析</span><span class="sxs-lookup"><span data-stu-id="cc8c1-103">Designing microservices: Domain analysis</span></span>

<span data-ttu-id="cc8c1-104">微服務最大的挑戰之一是如何定義個別服務之間的界限。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-104">One of the biggest challenges of microservices is to define the boundaries of individual services.</span></span> <span data-ttu-id="cc8c1-105">一般來說，一種服務應該「只做一件事」，但實際情況下這未必適用。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-105">The general rule is that a service should do "one thing" &mdash; but putting that rule into practice requires careful thought.</span></span> <span data-ttu-id="cc8c1-106">並沒有機械化的流程可以產生「正確的」設計。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-106">There is no mechanical process that will produce the "right" design.</span></span> <span data-ttu-id="cc8c1-107">您必須深入思考您的業務領域、需求和目標。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-107">You have to think deeply about your business domain, requirements, and goals.</span></span> <span data-ttu-id="cc8c1-108">否則，您的設計可能會變成一場災難，表現出某些不理想的特性，例如在服務之間存有隱藏的相依性、彼此密不可分，或介面的設計不良。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-108">Otherwise, you can end up with a haphazard design that exhibits some undesirable characteristics, such as hidden dependencies between services, tight coupling, or poorly designed interfaces.</span></span> <span data-ttu-id="cc8c1-109">在本章中，我們會以領域導向的方法來設計微服務。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-109">In this chapter, we take a domain-driven approach to designing microservices.</span></span>

<span data-ttu-id="cc8c1-110">微服務應該依照商務功能來設計，而非依照水平層 (例如資料存取或傳訊)。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-110">Microservices should be designed around business capabilities, not horizontal layers such as data access or messaging.</span></span> <span data-ttu-id="cc8c1-111">此外，微服務應該具有鬆散地結合和高度功能一致性的特性。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-111">In addition, they should have loose coupling and high functional cohesion.</span></span> <span data-ttu-id="cc8c1-112">若您可以變更一項服務，而不需要同時更新其他服務，就表示微服務是「鬆散地結合」。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-112">Microservices are *loosely coupled* if you can change one service without requiring other services to be updated at the same time.</span></span> <span data-ttu-id="cc8c1-113">若微服務具有單一、明確定義的用途 (例如管理使用者帳戶或追蹤遞送記錄)，就表示微服務具有「一致性」。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-113">A microservice is *cohesive* if it has a single, well-defined purpose, such as managing user accounts or tracking delivery history.</span></span> <span data-ttu-id="cc8c1-114">服務應納入領域知識，並從用戶端摘錄該知識。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-114">A service should encapsulate domain knowledge and abstract that knowledge from clients.</span></span> <span data-ttu-id="cc8c1-115">例如，即使用戶端不知道排程演算法的詳細資訊或管理無人機大隊的方式，也應該能夠排程無人機。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-115">For example, a client should be able to schedule a drone without knowing the details of the scheduling algorithm or how the drone fleet is managed.</span></span>

<span data-ttu-id="cc8c1-116">領域導向的設計 (DDD) 可提供一套架構，讓您設計良好的微服務。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-116">Domain-driven design (DDD) provides a framework that can get you most of the way to a set of well-designed microservices.</span></span> <span data-ttu-id="cc8c1-117">DDD 有兩個不同的階段：策略階段和戰術階段。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-117">DDD has two distinct phases, strategic and tactical.</span></span> <span data-ttu-id="cc8c1-118">在策略性 DDD 中，您會定義系統的大規模結構。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-118">In strategic DDD, you are defining the large-scale structure of the system.</span></span> <span data-ttu-id="cc8c1-119">策略性 DDD 有助於確保您的架構持續聚焦在商務功能上。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-119">Strategic DDD helps to ensure that your architecture remains focused on business capabilities.</span></span> <span data-ttu-id="cc8c1-120">戰術性 DDD 會提供一套可用來建立領域模型的設計模式。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-120">Tactical DDD provides a set of design patterns that you can use to create the domain model.</span></span> <span data-ttu-id="cc8c1-121">這些模式包括實體、彙總和領域服務。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-121">These patterns include entities, aggregates, and domain services.</span></span> <span data-ttu-id="cc8c1-122">這些戰術性模式可協助您設計既是鬆散地結合，又具有一致性的微服務。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-122">These tactical patterns will help you to design microservices that are both loosely coupled and cohesive.</span></span>

<span data-ttu-id="cc8c1-124">在這一章和下一章中，我們將逐步進行下列步驟，將其套用到無人機遞送應用程式：</span><span class="sxs-lookup"><span data-stu-id="cc8c1-124">In this chapter and the next, we'll walk through the following steps, applying them to the Drone Delivery application:</span></span>

## <a name="analyze-the-domain"></a><span data-ttu-id="cc8c1-140">分析領域</span><span class="sxs-lookup"><span data-stu-id="cc8c1-140">Analyze the domain</span></span>

## <a name="define-bounded-contexts"></a><span data-ttu-id="cc8c1-178">定義限界內容</span><span class="sxs-lookup"><span data-stu-id="cc8c1-178">Define bounded contexts</span></span>

<span data-ttu-id="cc8c1-179">領域模型會包括真實世界中事物的代表 &mdash; 使用者、無人機、包裹等等。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-179">The domain model will include representations of real things in the world &mdash; users, drones, packages, and so forth.</span></span> <span data-ttu-id="cc8c1-180">但這並不表示針對相同的事物，系統的每個部分都需要使用相同的代表。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-180">But that doesn't mean that every part of the system needs to use the same representations for the same things.</span></span>

<span data-ttu-id="cc8c1-181">例如，處理無人機修復和預測性分析的子系統會需要表示許多具有實體特性 (例如維修記錄、使用里程、壽命、型號、效能特性等等) 的無人機。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-181">For example, subsystems that handle drone repair and predictive analysis will need to represent many physical characteristics drones, such as their maintenance history, mileage, age, model number, performance characteristics, and so on.</span></span> <span data-ttu-id="cc8c1-182">但在排程遞送時，我們不會在意這些特性。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-182">But when it's time to schedule a delivery, we don't care about those things.</span></span> <span data-ttu-id="cc8c1-183">正在排程的子系統只需要知道無人機是否可用，以及收件和遞送的預計到達時間。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-183">The scheduling subsystem only needs to know whether a drone is available, and the ETA for pickup and delivery.</span></span>

<span data-ttu-id="cc8c1-184">若我們嘗試針對這兩個子系統建立單一模型，該模型會過於複雜。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-184">If we tried to create a single model for both of these subsystems, it would be unnecessarily complex.</span></span> <span data-ttu-id="cc8c1-185">這也會讓模型難以隨著時間而進化，因為任何變更都需要能夠讓在不同子系統上工作的多個小組滿意。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-185">It would also become harder for the model to evolve over time, because any changes will need to satisfy multiple teams working on separate subsystems.</span></span> <span data-ttu-id="cc8c1-186">因此，設計不同的模型，來代表兩個不同內容中的同一個真實世界實體 (在此案例中是無人機)，通常是較好的方式。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-186">Therefore, it's often better to design separate models that represent the same real-world entity (in this case, a drone) in two different contexts.</span></span> <span data-ttu-id="cc8c1-187">每一個模型都只包含在其特定內容中相關的功能和屬性。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-187">Each model contains only the features and attributes that are relevant within its particular context.</span></span>

<span data-ttu-id="cc8c1-195">在《Domain Driven Design (領域導向的設計)》一書中，Eric Evans 描述了在與其他限界內容互動時，數種維護領域模型完整性的模式。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-195">In the book *Domain Driven Design*, Eric Evans describes several patterns for maintaining the integrity of a domain model when it interacts with another bounded context.</span></span> <span data-ttu-id="cc8c1-196">其中一個微服務的主要原則是，服務會透過妥善定義的 API 進行通訊。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-196">One of the main principles of microservices is that services communicate through well-defined APIs.</span></span> <span data-ttu-id="cc8c1-197">此方法可對應到 Evans 稱為「開放主機服務」和「發行的語言」的兩種模式。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-197">This approach corresponds to two patterns that Evans calls Open Host Service and Published Language.</span></span> <span data-ttu-id="cc8c1-198">開放主機服務的概念是子系統會為了與其通訊的其他子系統，定義正式的通訊協定 (API)。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-198">The idea of Open Host Service is that a subsystem defines a formal protocol (API) for other subsystems to communicate with it.</span></span> <span data-ttu-id="cc8c1-199">發行的語言則延伸了此概念，以其他小組可用以寫入用戶端的形式來發行 API。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-199">Published Language extends this idea by publishing the API in a form that other teams can use to write clients.</span></span> <span data-ttu-id="cc8c1-200">在 [API 設計](./api-design.md)一章中，我們討論了使用 [OpenAPI 規格](https://www.openapis.org/specification/repo) (先前稱為 Swagger)，針對 REST API 定義適用於各種語言的介面描述，並以 JSON 或 YAML 格式表示。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-200">In the chapter on [API Design](./api-design.md), we discuss using [OpenAPI Specification](https://www.openapis.org/specification/repo) (formerly known as Swagger) to define language-agnostic interface descriptions for REST APIs, expressed in JSON or YAML format.</span></span>

## <a name="tactical-ddd"></a><span data-ttu-id="cc8c1-202">戰術性 DDD</span><span class="sxs-lookup"><span data-stu-id="cc8c1-202">Tactical DDD</span></span>

<span data-ttu-id="cc8c1-203">在 DDD 的策略性階段期間，您會對應業務領域並為領域模型定義限界內容。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-203">During the strategic phase of DDD, you are mapping out the business domain and defining bounded contexts for your domain models.</span></span> <span data-ttu-id="cc8c1-204">在戰術性 DDD 中則會更加精確地定義領域模型。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-204">Tactical DDD is when you define your domain models with more precision.</span></span> <span data-ttu-id="cc8c1-205">戰術性模式會在單一限界內容中套用。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-205">The tactical patterns are applied within a single bounded context.</span></span> <span data-ttu-id="cc8c1-206">在微服務架構中，我們對實體和彙總模式特別感興趣。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-206">In a microservices architecture, we are particularly interested in the entity and aggregate patterns.</span></span> <span data-ttu-id="cc8c1-207">套用這些模式可協助我們識別應用程式中服務的自然界限 (請參閱[下一章](./microservice-boundaries.md))。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-207">Applying these patterns will help us to identify natural boundaries for the services in our application (see [next chapter](./microservice-boundaries.md)).</span></span> <span data-ttu-id="cc8c1-208">一般來說，微服務應大於彙總，且小於限界內容。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-208">As a general principle, a microservice should be no smaller than an aggregate, and no larger than a bounded context.</span></span> <span data-ttu-id="cc8c1-209">首先，我們會檢閱戰術性模式。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-209">First, we'll review the tactical patterns.</span></span> <span data-ttu-id="cc8c1-210">然後我們會將其套用到無人機遞送應用程式中的「出貨」限界內容。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-210">Then we'll apply them to the Shipping bounded context in the Drone Delivery application.</span></span>

### <a name="overview-of-the-tactical-patterns"></a><span data-ttu-id="cc8c1-211">戰術性模式概觀</span><span class="sxs-lookup"><span data-stu-id="cc8c1-211">Overview of the tactical patterns</span></span>

<span data-ttu-id="cc8c1-212">本節提供了戰術性 DDD 模式的簡短摘要，因此若您已很熟悉 DDD，可以略過本節。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-212">This section provides a brief summary of the tactical DDD patterns, so if you are already familiar with DDD, you can probably skip this section.</span></span> <span data-ttu-id="cc8c1-213">模式在 Eric Evan 書中的第 5 &ndash; 6 章中，以及 Vaughn Vernon 所著的《Implementing Domain-Driven Design (實作領域導向的設計)》一書中會詳加說明。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-213">The patterns are described in more detail in chapters 5 &ndash; 6 of Eric Evans' book, and in *Implementing Domain-Driven Design* by Vaughn Vernon.</span></span>

<span data-ttu-id="cc8c1-215">**實體**。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-215">**Entities**.</span></span> <span data-ttu-id="cc8c1-216">實體是具有唯一的身分識別，且會持續一段時間的物件。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-216">An entity is an object with a unique identity that persists over time.</span></span> <span data-ttu-id="cc8c1-217">例如，在銀行應用程式中，客戶和帳戶會是實體。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-217">For example, in a banking application, customers and accounts would be entities.</span></span>

- <span data-ttu-id="cc8c1-218">實體在系統中具有唯一識別碼 (可用來查看或擷取實體)。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-218">An entity has a unique identifier in the system, which can be used to look up or retrieve the entity.</span></span> <span data-ttu-id="cc8c1-219">這並不表示識別碼一律會直接公開給使用者。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-219">That doesn't mean the identifier is always exposed directly to users.</span></span> <span data-ttu-id="cc8c1-220">而可能是資料庫中的 GUID 或主索引鍵。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-220">It could be a GUID or a primary key in a database.</span></span>
- <span data-ttu-id="cc8c1-221">身分識別可能會跨越多個限界內容，且持續時間可能會超過應用程式的存留期。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-221">An identity may span multiple bounded contexts, and may endure beyond the lifetime of the application.</span></span> <span data-ttu-id="cc8c1-222">例如，銀行帳戶號碼或政府所發行的 ID 就不會繫結到特定應用程式的存留期。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-222">For example, bank account numbers or government-issued IDs are not tied to the lifetime of a particular application.</span></span>
- <span data-ttu-id="cc8c1-223">實體的屬性可能會隨著時間變更。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-223">The attributes of an entity may change over time.</span></span> <span data-ttu-id="cc8c1-224">例如，某人的姓名或地址可能會變更，但他仍是同一個人。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-224">For example, a person's name or address might change, but they are still the same person.</span></span>
- <span data-ttu-id="cc8c1-225">實體可以擁有其他實體的參考。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-225">An entity can hold references to other entities.</span></span>

<span data-ttu-id="cc8c1-226">**值物件**。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-226">**Value objects**.</span></span> <span data-ttu-id="cc8c1-227">值物件沒有任何身分識別。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-227">A value object has no identity.</span></span> <span data-ttu-id="cc8c1-228">其只會由其屬性的值所定義。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-228">It is defined only by the values of its attributes.</span></span> <span data-ttu-id="cc8c1-229">值物件也是不可變的。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-229">Value objects are also immutable.</span></span> <span data-ttu-id="cc8c1-230">若要更新值物件，請一律要建立新執行個體來取代舊執行個體。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-230">To update a value object, you always create a new instance to replace the old one.</span></span> <span data-ttu-id="cc8c1-231">值物件可以具有封裝領域邏輯的方法，但這些方法對物件狀態必須沒有副作用。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-231">Value objects can have methods that encapsulate domain logic, but those methods should have no side-effects on the object's state.</span></span> <span data-ttu-id="cc8c1-232">值物件的典型範例包括色彩、日期和時間，以及貨幣值。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-232">Typical examples of value objects include colors, dates and times, and currency values.</span></span>

<span data-ttu-id="cc8c1-233">**彙總**。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-233">**Aggregates**.</span></span> <span data-ttu-id="cc8c1-234">彙總定義了一或多個實體周圍的一致性界限。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-234">An aggregate defines a consistency boundary around one or more entities.</span></span> <span data-ttu-id="cc8c1-235">彙總中只有一個的實體是根目錄。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-235">Exactly one entity in an aggregate is the root.</span></span> <span data-ttu-id="cc8c1-236">使用根目錄實體的識別碼即可查看。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-236">Lookup is done using the root entity's identifier.</span></span> <span data-ttu-id="cc8c1-237">彙總內的任何其他實體都是根目錄的子項目，並可透過追隨根目錄的指標加以參考。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-237">Any other entities in the aggregate are children of the root, and are referenced by following pointers from the root.</span></span>

<span data-ttu-id="cc8c1-238">彙總的目的是將交易的不變項目模型化。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-238">The purpose of an aggregate is to model transactional invariants.</span></span> <span data-ttu-id="cc8c1-239">真實世界中的事物關係錯綜複雜。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-239">Things in the real world have complex webs of relationships.</span></span> <span data-ttu-id="cc8c1-240">客戶會建立訂單、訂單會包含產品、產品會有供應商，依此類推。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-240">Customers create orders, orders contain products, products have suppliers, and so on.</span></span> <span data-ttu-id="cc8c1-241">若應用程式修改了數個相關的物件，要如何保證一致性？</span><span class="sxs-lookup"><span data-stu-id="cc8c1-241">If the application modifies several related objects, how does it guarantee consistency?</span></span> <span data-ttu-id="cc8c1-242">我們要如何追蹤不變項目並加以強制執行？</span><span class="sxs-lookup"><span data-stu-id="cc8c1-242">How do we keep track of invariants and enforce them?</span></span>  

<span data-ttu-id="cc8c1-243">以前傳統應用程式通常會使用資料庫交易來強制確保一致性。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-243">Traditional applications have often used database transactions to enforce consistency.</span></span> <span data-ttu-id="cc8c1-244">然而，在分散式應用程式中，這通常不可行。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-244">In a distributed application, however, that's often not feasible.</span></span> <span data-ttu-id="cc8c1-245">單一商業交易可能會跨越多個資料存放區、可能會長時間執行，或可能涉及第三方服務。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-245">A single business transaction may span multiple data stores, or may be long running, or may involve third-party services.</span></span> <span data-ttu-id="cc8c1-246">最後是取決於應用程式 (而非資料層) 來強制執行領域所需的不變項目。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-246">Ultimately it's up to the application, not the data layer, to enforce the invariants required for the domain.</span></span> <span data-ttu-id="cc8c1-247">這就是彙總要模型化的項目。</span><span class="sxs-lookup"><span data-stu-id="cc8c1-247">That's what aggregates are meant to model.</span></span>
